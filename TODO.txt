How the static files get served (verified)
------------------------------------------
- DocumentRoot is `/srv/www/zfin.org/home` (docker/httpd/conf-zfin.org). In the
  httpd container `/srv/www` -> `/opt/zfin/www_homes` (Dockerfile symlink) and the
  `www_data` volume is mounted there, so DocumentRoot == $TARGETROOT/home.
- `gradle make` copies home/zf_info/** (SimpleDirectoryCopyTask, build.gradle) and
  server_apps/apache/** into $TARGETROOT. So `gradle make` is what deploys both the
  generated pages AND this inc-redirect change. (Static assets deploy via `make`,
  NOT `dirtydeploy`; the Tomcat webapp copy excludes zf_info/**, build.gradle.)
- There is NO `ProxyPass /zf_info`. Requests reach Tomcat only via the inc-redirect
  rewrite below. The guard is therefore the actual switch that makes Apache serve a
  converted page: file present -> Apache static; file absent -> Tomcat.
- GOTCHA: the guard MUST be `RewriteCond %{DOCUMENT_ROOT}%{REQUEST_URI} !-f`, NOT
  `%{REQUEST_FILENAME} !-f`. In VirtualHost context REQUEST_FILENAME is not mapped
  to a filesystem path (it equals REQUEST_URI), so a bare `!-f` never matches and
  the rewrite always fires -> everything keeps going to Tomcat.
- RELOAD: `gradle make` deploys the file but Apache reads it at startup. Reload with
  `zexec httpd -c 'httpd -k graceful'` (apachectl/systemctl not available in the
  container). A plain `make` alone will NOT pick up inc-redirect changes.

================================================================================
DESIGN NOTE: moving the static content (static/) into its own repository
================================================================================

Status: NOT done -- design/plan only. Decide whether it's worth it (see "Is it
worth it?" at the end) before implementing.

Goal
----
Move the served static content -- static/ (zf_info reference pages, images, ZFIN,
robots.txt, favicon.ico, analytics.js) -- into a separate git repo (e.g.
"zfin-static") with its own deploy, so archival content can be versioned/published
independently of the main app.

Why this is feasible (the hard decoupling is already done)
----------------------------------------------------------
- Static pages reference stable /dist/*.latest.{css,js} webpack aliases (LatestAlias
  Plugin), not content-hashed names -> app rebuilds don't invalidate them.
- Header/footer are injected client-side from /action/layout/header|footer at runtime
  (zfin-chrome.js), not baked into the pages.
- analytics.js is a self-contained static host-switched file (no build substitution).

What stays COUPLED (treat as a versioned contract between the two repos)
------------------------------------------------------------------------
The pages are NOT standalone; at runtime, from the same origin, they need the MAIN
app to keep serving:
  - /dist/style.latest.css, /dist/vendor-common.latest.js, /dist/zfin-common.latest.js
  - /action/layout/header  and  /action/layout/footer   (Tomcat / ChromeController)
Keep those alias names and endpoints stable, or coordinate changes across the repos.

New repo (zfin-static)
----------------------
Contents: the static/ tree + convert-zfbook-to-static.groovy (the one-shot converter).

Main-repo changes (small + bounded; these are ALL the references today)
-----------------------------------------------------------------------
- build.gradle: delete static/ and the 4 deploy tasks that copy it into
  $TARGETROOT/home: home;root;deployFiles (robots/favicon/analytics), home;images;
  deployFiles, home;zf_info;deployFiles, home;ZFIN;deployFiles.
- GenerateSitemapTask.java: it currently scans SOURCEROOT/static/zf_info. Repoint it
  to scan the DEPLOYED tree WEBROOT_DIRECTORY/zf_info ($TARGETROOT/home/zf_info) so the
  sitemap enumerates what's actually served and needs no static-repo checkout. (Also
  more correct in general.)
- KEEP in the app (serving mechanism, not content): server_apps/apache/inc-redirect,
  docker/httpd/conf-zfin.org, ChromeController + home/WEB-INF/jsp/layout/chrome-*.jsp,
  frontend/css/layout.scss (header height reserve), the webpack .latest aliases.

Deploying the static repo (must land files at $TARGETROOT/home/{zf_info,images,ZFIN,
...} = Apache DocumentRoot on the www_data volume). Options, simplest first:
  1. A lightweight Jenkins/CI job that checks out zfin-static and rsyncs it into
     $TARGETROOT/home, triggered by that repo's webhook. Independent cadence, ZERO
     Apache changes. RECOMMENDED starting point (fits existing Jenkins).
  2. Populate the www_data volume from a small image / init step at deploy time.
  3. Separate origin / CDN (object store + CDN, or a second Apache Alias/vhost) --
     biggest change; only if you want content off the app servers entirely.

Coordination + ergonomics
--------------------------
- Two pipelines then write to $TARGETROOT/home but to DISJOINT subtrees: the app owns
  /dist, WEB-INF, etc.; zfin-static owns zf_info/, images/, ZFIN/ (analytics.js moves
  with it). Document the ownership so they don't clobber each other.
- Dev ergonomics (a local app needs the content to view those pages): use a git
  submodule at static/, or a sibling checkout mounted by the dev docker-compose, or
  let the same deploy job populate dev.

Is it worth it?
---------------
This is archival content that changes rarely, so the independent-deploy payoff is
modest, and you take on a second pipeline + the cross-repo .latest.*//action/layout
contract. Do it if non-devs will edit the content or you need to publish content
without an app release; otherwise the current in-repo static/ (already cleanly split
in the tree) is likely the better cost/benefit.

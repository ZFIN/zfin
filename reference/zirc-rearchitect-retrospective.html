Rearchitecting the ZIRC submission form

A retrospective on rebuilding the ZIRC line-submission editor from a per-section React-Hook-Form layout into a schema-driven, JSON-Schema-first system with per-field autosave, a DB-backed audit trail, multipart attachments, and an OpenAPI 3 spec. Covers M1 through M4.3 plus the polish + OpenAPI tail. Written for a ZFIN dev who already knows Java/Spring + TypeScript but hasn't worked with Zod / React Hook Form / rjsf / JSON Forms before.

TL;DR

  • Single source of truth is Java. The server ships both the JSON Schema and the JSON Forms uiSchema; the React form is a thin renderer with no hand-coded field list.
  • Writes are per-field, not per-form. PATCH endpoints take { path, value }. Audit log captures one row per leaf change; lost-update races between sibling fields are prevented with Hibernate @DynamicUpdate.
  • We chose JSON Forms over rjsf after a head-to-head spike. JSON Forms' "uiSchema dispatches to renderers ranked by tester functions" model fit our reference-styled markup better than rjsf's prop-soup widget overrides.
  • Per-aggregate flat paths (one PATCH endpoint per row id: submission, mutation, assay) let us avoid a JSON-pointer path resolver. Nested aggregates get their own card with their own form scope.
  • Audit, undo, and validation all become easier when you treat the form as an event stream of field changes rather than a snapshot of the document.

Where we started

The starting state on zirc-rearchitect (commit 5ca8ea9fa9, branched from PR #1845's tip) was almost a blank slate:

  • Backend entities and DB schema — LineSubmission, Mutation, GenotypingAssay, GenotypingAssayFile, the 1182 Liquibase migrations — already landed from PR #1845.
  • JSP shell with an empty React mount — line-submission-edit.jsp had <div class="__react-root" id="LineSubmissionEdit"> with nothing to mount. No LineSubmissionEdit.tsx, no form code, no DTO, no service.
  • Detail page with inline editing — line-submission-detail.jsp used editable-field spans (the trunk-style "click a span, it turns into an input, save on blur" idiom).

So the rearchitect wasn't replacing a prior form architecture — it was building the edit experience from a clean slate, on top of an existing entity/DB layer.

The alternate-universe version. A parallel implementation for the same blank slate (ZFIN-10265 / PR #1864) landed on main a day before this branch's starting marker. It lives at zfin-10265-zirc-line-submission-react-re-architect-squash. That version built a hand-rolled form renderer (section()/field()/array()/custom() schema builders with a useAutosavedForm hook + per-path PATCH + a SaveToast) plus a LineSubmissionDTO and LineSubmissionService. Some of the things this retrospective treats as "what we built" — per-field PATCH, save-on-blur, the assay-type field matrix idea, even the save toast — already existed in that parallel branch. Our distinguishing bet was schema-as-data published by the server (JSON Forms + uiSchema rules) instead of schema-as-code authored in the client.

What we wanted from the rebuild, beyond "fill in the empty React mount":

  • Per-field audit trail. Curators wanted to know who changed which field and when. Section-scoped writes can't distinguish "you renamed the line" from "you re-saved the section while you were there."
  • No cross-field lost updates. Two tabs writing the same row would clobber each other's columns even when they touched different fields, because Hibernate's default UPDATE writes every column.
  • One source of truth for shape and constraints. The Java entity, any client-side schema, the JSP, and the reference HTML otherwise drift on field names, max lengths, and required-ness. The hand-rolled-FormRenderer path (ZFIN-10265) put the schema in TypeScript; we wanted it in Java so it couldn't get out of sync with the entity.

A quick library glossary

Zod (TypeScript)

Runtime schema validator. You write z.object({ name: z.string().max(255) }) and it gives you both the validation function and a TypeScript type via z.infer. We used it heavily in M2, then removed it once the server took over as the source of truth for shape and constraints. Mentioned here so you can recognize schema.parse(input) in older code.

React Hook Form (RHF)

Form state library. Tracks dirty/touched/error per field, integrates with Zod via @hookform/resolvers/zod. Lives at the field component level — each <input> is registered with {...register("name")}, and the whole form has a handleSubmit. M2 forms used RHF; M3+ forms don't.

rjsf (@rjsf/core)

"React JSON Schema Form." Takes a JSON Schema, renders a form. You override per-widget templates (string widget, array template, field template, error template) to control markup. The model is "one fat <Form> component plus a forest of overridable templates."

JSON Forms (@jsonforms/react)

Different model. You pass two schemas — the JSON Schema for shape, and a separate uiSchema describing layout and per-control widget hints. The runtime dispatches each uiSchema element to whichever renderer has the highest-ranked tester match. You don't override templates; you register renderers. We picked this one.

React Query (@tanstack/react-query)

Server-state cache for React. useQuery for fetches, useMutation for writes, automatic invalidation by key. We use it as the only source of server data; React component state is purely local (form drafts, expansion state, etc.).

M1 / M2: per-section autosave with Zod + RHF

Filling the empty React mount, our first cut split the form into four sections and built each with React Hook Form + Zod. Each section had its own component, its own Zod schema, and its own autosave hook. The save endpoint was section-scoped: POST /api/zirc/line-submissions/{id}/overview with the section's whole subdocument as the body. 

// home/javascript/react/zirc/hooks/useSectionAutosave.ts (M2 era)
export function useSectionAutosave<T extends FieldValues>({
  schema, defaultValues, save,
}: Options<T>) {
  const form = useForm<T>({
    resolver: zodResolver(schema),
    defaultValues,
    mode: "onChange",
  });
  const watched = form.watch();
  const key = JSON.stringify(watched);
  React.useEffect(() => {
    if (!form.formState.isDirty || !form.formState.isValid) return;
    const handle = setTimeout(() => save(watched), 800);
    return () => clearTimeout(handle);
  }, [key]);
  return form;
}

This worked, but three limits showed up fast:

  1. Field-level audit is awkward. Section-scoped writes can't distinguish "user changed the line name" from "user re-saved an unchanged section."
  2. Schema drift was still there. The Zod schema in TS, Jakarta Bean Validation in Java, and the entity column lengths in SQL could all disagree.
  3. Parallel evidence. ZFIN-10265 had already arrived at per-field PATCH the hard way — through a hand-rolled useAutosavedForm hook with a save queue, path-keyed commits, and DTO-shaped save responses. That branch's existence was a useful signal: per-field is the right granularity, and we should adopt it without re-inventing the renderer.

The pivot: schema-driven forms

Two pivots at once, looking sideways at the alternate-universe branch:

  1. Per-field, not per-section. Both the alt branch and our own M1/M2 lessons pointed here.
  2. Server-published schema instead of client-authored. This is where we diverge from ZFIN-10265. They authored the schema in TypeScript with a builder DSL; we publish it from Java as JSON Schema + a JSON Forms uiSchema.

The result: the server owns the form schema. The React app fetches { schema, uiSchema } from GET /api/zirc/form-schema and just renders it. The client doesn't know what fields exist until the server tells it.

Writes flow the other way: the form emits one PATCH per leaf change, with a JSON-pointer path and the new value: 

PATCH /api/zirc/line-submissions/ZDB-LINESUBMISSION-260515-7
Content-Type: application/json

{ "path": "/acceptance/reasons", "value": ["expect_high_demand", "interesting_gene"] }

On the server, the path is looked up in a FIELDS map of FieldDescriptor records — one read+write closure per legal path: 

public record FieldDescriptor(
    Function<LineSubmission, JsonNode> read,
    BiConsumer<LineSubmission, JsonNode> write) {}

public static final Map<String, FieldDescriptor> FIELDS = Map.ofEntries(
    field("/name",
        LineSubmission::getName,
        (s, v) -> s.setName(text(v))),
    field("/acceptance/reasons",
        LineSubmission::getReasons,
        (s, v) -> s.setReasons(stringArray(v))),
    // ...
);

This single map gives us four things at once:

  • Authorization gate. Unknown paths are rejected at the controller — clients can't write columns the form doesn't expose.
  • Audit log. The same descriptor's read captures the pre-update value, so old/new round-trip through Jackson consistently.
  • Persistence. The write closure applies the typed coercion (trim, null-on-blank, etc.) before Hibernate sees the entity.
  • Schema generation. The same file builds the JSON Schema sent to the client. One file, one matrix.

The service method is then almost boring, which is the point:

public LineSubmission updateField(String zdbID, FieldUpdate update) {
    LineSubmission submission = getRequiredLineSubmission(zdbID);
    FieldDescriptor d = ZircFormSchema.FIELDS.get(update.path());
    if (d == null) throw new IllegalArgumentException("Unknown path: " + update.path());

    JsonNode oldValue = d.read().apply(submission);
    HibernateUtil.createTransaction();
    d.write().accept(submission, update.value());
    HibernateUtil.flushAndCommitCurrentSession();

    log.info("ZIRC_AUDIT user={} submission={} path={} old={} new={}",
        currentUserId(), zdbID, update.path(), safeJson(oldValue), safeJson(update.value()));
    return submission;
}

The deciding factor was the conditional-reveal idiom. JSON Forms lets you write:

// In Java, emitted to the client as part of uiSchema
Map<String, Object> showWhenLethal = Map.of(
    "effect", "SHOW",
    "condition", Map.of(
        "scope",  "#/properties/homozygousLethal",
        "schema", Map.of("const", true)));

group("Lethality", List.of(
    controlWithOptions("#/properties/homozygousLethal", Map.of("widget", "yesNoRadio")),
    controlWithRule("#/properties/lethalityStageTypical", Map.of(), showWhenLethal),
    controlWithRule("#/properties/lethalityWindowStart",  Map.of(), showWhenLethal),
    controlWithRule("#/properties/lethalityAdditionalInfo", Map.of("multi", true), showWhenLethal)
));

For the assay-type field matrix (M4.2), this scales to schema.enum membership tests — "show this group when assayType is one of {PCR, RFLP, dCAPS, sequencing, KASP}":

private static Map<String, Object> groupWithRule(
    String label, List<String> visibleForAssayTypes, List<Map<String, Object>> elements) {
    Map<String, Object> g = group(label, elements);
    g.put("rule", Map.of(
        "effect", "SHOW",
        "condition", Map.of(
            "scope",  "#/properties/assayType",
            "schema", Map.of("enum", visibleForAssayTypes))));
    return g;
}

The autosave loop on the client is small enough to paste in full:

React.useEffect(() => {
    // Guard: don't run until the seed effect has populated formData.
    if (formData == null || lastSavedRef.current == null) return;

    const handle = window.setTimeout(async () => {
        const changes = diffLeaves(lastSavedRef.current, formData)
            .filter(([path]) => !EXTERNALLY_MANAGED_PATHS.has(path));
        if (changes.length === 0) return;

        setStatus("saving");
        try {
            for (const [path, value] of changes) {
                await api.patch(`/assays/${assayId}`, { path, value });
            }
            lastSavedRef.current = formData;
            setStatus("saved");
        } catch (e) {
            setStatus("error");
        }
    }, AUTOSAVE_DEBOUNCE_MS);

    return () => window.clearTimeout(handle);
}, [formDataKey]);

diffLeaves is a 25-line recursive walk. Arrays are treated as leaves (the whole list is one value), which matches how the reasons chip-list and publications list are modeled on the server.

M3 / M4: scaling to mutations and assays

The submission, the mutation, and the assay are all separate aggregates with their own row id, their own schema endpoint, their own PATCH endpoint, and their own flat-path namespace:

AggregateSchema endpointField PATCH endpointFIELDS map
LineSubmissionGET /api/zirc/form-schemaPATCH /api/zirc/line-submissions/{zdbID}ZircFormSchema.FIELDS
MutationGET /api/zirc/mutations/form-schemaPATCH /api/zirc/mutations/{id}ZircMutationFormSchema.FIELDS
GenotypingAssayGET /api/zirc/assays/form-schemaPATCH /api/zirc/assays/{id}ZircAssayFormSchema.FIELDS

Child lists (mutations on the submission, assays on the mutation) are server-managed from the parent form's point of view. The parent schema includes them as summary rows, but Add/Delete go through dedicated POST/DELETE endpoints — not the field-path PATCH. The parent form's autosave diff filters those paths out so a list mutation doesn't fire a spurious "PATCH the whole array" attempt: 

const EXTERNALLY_MANAGED_PATHS = new Set<string>(["/mutations"]);
// (MutationEdit has its own: ["/assays"])

const changes = diffLeaves(lastSavedRef.current, formData)
    .filter(([path]) => !EXTERNALLY_MANAGED_PATHS.has(path));

In M4.2 we deliberately mounted the per-assay editor inline inside an expanded card rather than navigating to a dedicated /zirc/assay/{id}/edit page. The mutation page picked the latter (own route), the assay layer picked the former (inline). Both work; the trade-off is:

  • Inline expansion. Faster UX, no navigation hop, but you have to be careful that the inline form's data flow doesn't entangle with the parent's. We solved this by giving each AssayEdit its own React Query and its own seed/autosave loop scoped to its own row id.
  • Own route. Cleaner state isolation (mount/unmount on navigation), but two clicks to reach a field.

Critically, both choices keep field paths flat within a single PATCH endpoint. The "what if we used /assays/0/forwardPrimer against the parent mutation?" path-resolver question stays deferred — and looks increasingly like it might never need to be answered.

M4.3: multipart attachments and the IllegalArgumentException gap

The genotyping_assay_file table started life with a four-kind matrix (chromatogram / gel_image / result_image / melt_curve) gated on assay type. Curators ultimately wanted one generic "Attachments" slot per assay instead — simpler today, with the option to re-introduce categorization later. The implementation reflects that:

  • Liquibase migration drops the af_kind CHECK constraint and makes the column nullable. The column itself stays so future categorization isn't blocked.
  • Multipart upload at POST /api/zirc/assays/{id}/attachments. The on-disk layout is server-constructed: $TARGETROOT/server_apps/data_transfer/ZIRC/<submission>/assay-<assay id>-<file id>-<sanitized filename> — the client never sees a path.
  • Streaming download at GET /api/zirc/assays/attachments/{id}/content. Content-Disposition: inline; filename="…" uses the original filename, not the on-disk name.
  • Guards: 20 MB cap, content-type allowlist (jpeg/png/gif/svg/tiff/pdf/plain/csv).

Two architectural points worth calling out:

1. Sanitize-before-store with an audit-log breadcrumb

// Strip path separators, leading dots, traversal, and length-cap.
static String sanitizeFilename(String name) {
    if (name == null) return "file";
    String trimmed = name.replaceAll("[\\\\/\\u0000-\\u001F]", "_");
    trimmed = trimmed.replace("..", "_");
    while (trimmed.startsWith(".")) trimmed = trimmed.substring(1);
    if (trimmed.isBlank()) return "file";
    if (trimmed.length() > 200) trimmed = trimmed.substring(0, 200);
    return trimmed;
}

The DB stores both af_original_filename (user-supplied) and af_stored_path (server-constructed). Audit log captures both at upload time and again at delete time, so even a deleted-then- removed-from-disk attachment leaves a queryable trail.

2. The IllegalArgumentException gap (caught during verification)

Symptom: Uploading a disallowed content type rejected the file (good) but returned a 500 + HTML error page instead of an RFC 7807 ProblemDetail.
Root cause: ZircApiExceptionHandler mapped ZircEntityNotFoundException and MethodArgumentNotValidException but not the bare IllegalArgumentException that the service throws for size and content-type rejections. Spring's default mapping for IllegalArgumentException is 500.
Fix: Add a 400 mapping. Important to notice: this gap also affected the existing "Unknown form field path" rejections in updateField/updateMutationField/updateAssayField — all three were silently returning HTML 500 pages instead of JSON. One small handler fixed all four call sites at once. 
@ExceptionHandler(IllegalArgumentException.class)
public ProblemDetail handleBadRequest(IllegalArgumentException e) {
    ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, e.getMessage());
    pd.setTitle("Bad Request");
    pd.setType(URI.create("https://zfin.org/problems/bad-request"));
    return pd;
}

Lesson: error mappers can have silent coverage gaps that don't show up until the unhappy path actually runs. Worth doing a "throw every kind of failure" sweep when you build out an exception advice — happy paths exercise it less than you'd think.

DB-backed audit trail

Through M4.2 the audit log was log4j-only — ZIRC_AUDIT lines in catalina.out. Curators wanting to query history had to ssh in. Replaced (additively — the log lines still emit) with a zirc.audit table: 

CREATE TABLE zirc.audit (
    ae_id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    ae_actor       TEXT NOT NULL,
    ae_entity_kind TEXT NOT NULL,        -- submission|mutation|assay
    ae_entity_id   TEXT NOT NULL,
    ae_action      TEXT NOT NULL,        -- update|create-*|delete-*|upload|delete-file
    ae_path        TEXT,                 -- JSON pointer for `update`, NULL otherwise
    ae_old_value   JSONB,
    ae_new_value   JSONB,
    ae_at          TIMESTAMP NOT NULL DEFAULT now()
);
CREATE INDEX audit_entity_idx ON zirc.audit (ae_entity_kind, ae_entity_id, ae_at);
CREATE INDEX audit_actor_idx  ON zirc.audit (ae_actor, ae_at);

The service-side change was a single helper that writes both the row and the log line, called from every existing audit site. Because the insert goes through the same Hibernate session as the main change, audit rows roll back automatically if the main commit fails — a property we didn't have to engineer; it just falls out of putting the insert before the flushAndCommitCurrentSession() call. 

private void writeAudit(String entityKind, String entityId, String action,
                        String path, JsonNode oldValue, JsonNode newValue) {
    Person currentUser = ProfileService.getCurrentSecurityUser();
    String actor = currentUser == null ? "anonymous" : currentUser.getZdbID();

    AuditEntry entry = new AuditEntry();
    entry.setActor(actor);
    entry.setEntityKind(entityKind);
    entry.setEntityId(entityId);
    entry.setAction(action);
    entry.setPath(path);
    entry.setOldValue(safeJson(oldValue));
    entry.setNewValue(safeJson(newValue));
    repository.save(entry);

    log.info("ZIRC_AUDIT user={} {}={} action={} path={} old={} new={}",
            actor, entityKind, entityId, action,
            path == null ? "-" : path, safeJson(oldValue), safeJson(newValue));
}

An interesting side effect: actions that weren't audited before (Add/Delete mutation, Add/Delete assay, upload, delete-file) are audited now because they all flow through the same helper. The audit table is the single source of truth and the log line is a reasonable shadow for any tooling that's still grepping logs.

OpenAPI 3 spec + drift test + Swagger UI

After the API stabilized at ~17 endpoints we wanted a single developer-readable contract. The natural choice would be springdoc-openapi v2, but it assumes Spring Boot autoconfig — we run plain Spring MVC. The decision matrix lived in reference/zirc-openapi-approach.md; TL;DR:

  • Picked: hand-curated YAML at home/WEB-INF/openapi/zirc-api.yaml, served by a small controller.
  • Not picked: springdoc-openapi (Boot-conditional bean wiring is brittle without Boot), swagger-core annotations + build-time generation (overkill at 17 endpoints).

Drift detection

The cost of hand-curating is that the YAML can rot. Mitigated with a JUnit 4 test that reflects over the four ZIRC controllers and asserts every (method, path) pair appears in both directions: 

// In the test:
for (Class<?> controller : CONTROLLERS) {
    String classPrefix = classRequestMapping(controller);
    for (Method method : controller.getDeclaredMethods()) {
        for (var e : METHOD_ANNOS.entrySet()) {           // GET/POST/PATCH/DELETE
            var anno = method.getAnnotation(e.getKey());
            if (anno == null) continue;
            for (String methodPath : pathsFor(anno)) {
                handlerEndpoints.add(e.getValue().toUpperCase() + " "
                        + stripApiZircPrefix(classPrefix + methodPath));
            }
        }
    }
}
// then: assertEquals(specEndpoints, handlerEndpoints)

The YAML is declared as a Gradle test input so spec-only changes invalidate the task and the test re-runs. Verified by tampering with the YAML and observing both asserts fire (one for missing handler, one for missing spec entry).

Why this test ended up in JUnit 4 instead of Spock

Worth flagging for future test-writers: the codebase has ~15 .groovy Spock specs sitting in test/, mentioned in the gradle test filter allowlist, that never actually run. Two reasons:

  1. The test source set declared java.srcDirs = ['test'] but no groovy srcDir. The Groovy plugin's default is test/groovy/, which doesn't exist in this layout. So compileTestGroovy was reporting NO-SOURCE and no Spock .class files were produced.
  2. Even with that fixed (added groovy { srcDirs = ['test'] }), Spock 2.x requires useJUnitPlatform() — which is commented out in build.gradle. So the specs would still need an opt-in to the JUnit Platform runner.

Fixing #1 was straightforward (one pre-existing dead Groovy file with a missing import had to be excluded). #2 would surface dozens of dormant specs at once with unknown breakage; the right scope is "deliberate Spock revival" rather than a side effect. Until then, default to JUnit 4 Java for new tests.

Swagger UI: vendor it

The /api/zirc/docs route renders a small HTML shell that mounts Swagger UI against the YAML. First cut loaded the swagger-ui-dist CSS and JS from jsdelivr; that worked but means the page is broken without internet and at the mercy of CDN outages. Replaced with the vendored files (~1.6 MB total) under home/WEB-INF/openapi/swagger-ui/, served via two extra controller routes. The drift test now also covers these routes so accidental removal breaks CI.

Pitfalls and how we resolved them

Cross-section lost updates (Hibernate default UPDATE writes all columns)

Symptom: Two near-simultaneous field-path PATCHes against the same row clobber each other's untouched columns on commit.
Fix: @DynamicUpdate on every entity that has field-path PATCH support. 
@Entity(name = "ZircLineSubmission")
@Table(schema = "zirc", name = "line_submission")
@DynamicUpdate   // ← critical
@Getter @Setter
public class LineSubmission { ... }

Spurious "clear field" PATCHes during slow page reload

Symptom: After saving multiple fields, reloading the page sometimes fired a PATCH clearing an unrelated field (e.g. path=/alleleDesignation old="zf-test-allele" new="").
Root cause: Race between useState's initial empty-defaults value (captured on first render before the GET resolved) and the seed useEffect's setFormData. The autosave effect could close over empty-defaults formData while lastSavedRef had already been updated to the seeded values; the diff then emitted "non-empty → empty" changes.
Fix: Type formData as T | null, render "Loading…" until the seed effect runs, early-return the autosave effect when either is null. 
const [formData, setFormData] = React.useState<FormDataShape | null>(null);
const lastSavedRef = React.useRef<FormDataShape | null>(null);

React.useEffect(() => {
    if (!mutation || formData !== null) return;
    const seed = initialDataFromMutation(mutation);
    setFormData(seed);
    lastSavedRef.current = seed;
}, [mutation?.id, formData]);

// ... in render
if (formData == null) return <p>Loading…</p>;

ProblemDetail handler not catching errors

Symptom: Validation errors came back as HTML 500 pages instead of RFC 7807 JSON.
Root cause: Spring picks the most specific @ExceptionHandler by exception type, but for sibling beans it picks by registration order. A pre-existing ZfinGlobalExceptionHandler.@ExceptionHandler(Exception.class) was matched first.
Fix: @Order(Ordered.HIGHEST_PRECEDENCE) on the ZIRC advice so it gets first crack.

Section rule visibility silently ignored (M4.2)

Symptom: Conditional reveal worked for individual Control elements but not for Group elements — PCR Primers showed even when assayType was empty.
Root cause: JSON Forms' built-in withJsonFormsControlProps evaluates the rule and passes props.visible. Our custom SectionRenderer uses withJsonFormsLayoutProps, which also passes visible, but we weren't reading it.
Fix: A single early-return: 
function SectionRenderer({ uischema, schema, path, enabled, visible, ... }: LayoutProps) {
    if (visible === false) return null;  // ← respects uiSchema rules
    ...
}

200 vs 204 on void DELETE

Symptom: Delete fired on the server (DB confirmed), but the React UI didn't update.
Root cause: Spring @DeleteMapping returning void defaults to 200 with empty body. Our api.delete<void> client did response.json() on a 200, which threw, so the onSuccess handler never ran — meaning React Query never invalidated.
Fix: @ResponseStatus(HttpStatus.NO_CONTENT) on every delete endpoint. The client already special-cases 204: 
if (response.status === 204) return undefined as unknown as T;
return response.json() as Promise<T>;

List didn't update after Add/Delete

Symptom: Add a mutation to a submission — the React Query cache refreshed, but the form still showed the old list.
Root cause: SchemaForm's formData is locally-owned for keystroke responsiveness; the list lives inside it. Invalidating the React Query cache refetched, but the local copy was stale.
Fix: Sync the server-managed slice into local state via a dependency on a JSON-stringified key: 
const mutationsKey = JSON.stringify(submission?.mutations ?? []);
React.useEffect(() => {
    if (!submission || formData === null) return;
    const next = submission.mutations ?? [];
    setFormData((d) => (d == null ? d : { ...d, mutations: next }));
    if (lastSavedRef.current != null) {
        lastSavedRef.current = { ...lastSavedRef.current, mutations: next };
    }
}, [mutationsKey]);

Spring property name collision (path-as-id)

Symptom: Nested form-data paths produced HTML IDs like fr-background.maternalBackground with literal dots in them, breaking document.getElementById in tests.
Fix: One-liner in the row renderer: const id = `fr-${path.split('.').pop()}`. The collision risk between leaves of different objects is real but hasn't bitten us yet.

Head-to-head: this branch vs ZFIN-10265

The starting-point section flagged that zfin-10265-zirc-line-submission-react-re-architect-squash is a parallel implementation of the same blank slate. With everything this branch built now on the table, here's a head-to-head on the axes where the two branches actually differ.

Schema authoring location

This branch — server-published JSON Schema + uiSchemaZFIN-10265 — TypeScript schema builders
Pros One source of truth in Java, where the entity already lives. Drift is essentially impossible — adding a column means editing FIELDS and the uiSchema in the same file. Curators can change enums and labels without a frontend deploy. TypeScript catches a missing path at compile time. Per-field metadata (helpText, infoHref, placeholder, suffix) is expressive without abusing JSON Forms options. Form renders immediately — no schema fetch.
Cons Pre-render flash: until the schema response lands, the form is just "Loading…". Schema builders are imperative LinkedHashMaps — forgetting to register a path that the uiSchema references blows up at runtime when the user types in the field. No compile-time cross-check between the schema and the FIELDS map. Drift surface: the Java entity and the TS schema can disagree on max lengths and required-ness without anything failing. New aggregates need parallel TS schema files; the server doesn't know what the client is showing. Curator-driven changes require a deploy.

Form rendering

This branch — JSON Forms with ~10 custom renderersZFIN-10265 — hand-rolled FormRenderer
Pros Battle-tested library; conditional reveal via uiSchema rule blocks is declarative and inherits accessibility, focus management, transitions for free. Adding new control types is a rankWith(20, ...) registration. No external forms-library dependency. Plain TypeScript schema is debuggable in any tooling that understands TS. Direct control over markup — no library-imposed structural choices to fight.
Cons We still wrote ~10 custom renderers because the default Bootstrap theme didn't match trunk markup; the "free library" still cost LOC. JSON Forms' tester/rank dispatch is obscure for newcomers — debugging "why is the default renderer winning" needs library familiarity. props.visible from rules works automatically for Controls but not for layouts (we hit this — SectionRenderer was silently ignoring rules until we added the early return). Every concern (conditional reveal, validation, focus, accessibility) has to be reimplemented. Rendering, validation, and layout are coupled in the same component — extending is more invasive than a library renderer registration. The renderer grows with the form; ours stays roughly constant.

Concurrency / lost-update prevention

This branch — Hibernate @DynamicUpdateZFIN-10265 — client save queue
Pros Solves the problem at the persistence layer — works no matter how many client tabs are open, no matter how the client batches writes. Zero client complexity. Serializes saves within one tab — no two PATCHes in flight simultaneously. Avoids out-of-order response problems where a later save's response races a prior save's. Reduces wire chatter by batching section writes.
Cons Per-field UPDATE statements (one PATCH = one UPDATE of one column) are chattier on the wire than batched writes. Forgetting @DynamicUpdate on a new entity silently re-introduces lost updates — no test catches it. Doesn't help across tabs/sessions — two curators editing the same row from different machines can still clobber each other. Adds careful but easy-to-break client state (in-flight queue, monotonic seq, dtoRef vs setDtoState mirror). The section-scoped writes that the batching enables also re-introduce the audit-granularity problem.

Honest take: @DynamicUpdate is structurally stronger. The save queue is a useful client-side optimization but doesn't replace it — they're complementary, not alternatives.

Endpoint shape and service complexity

This branch — per-aggregate field-path PATCHZFIN-10265 — domain-specific save methods
Shape 17 endpoints across 4 controllers. ~400-line ZircSubmissionService. One write path per aggregate; new fields plug into a FIELDS map. ~25 endpoints in one controller. 844-line LineSubmissionService. Mix of per-field (/patch) and per-section (saveReasons, saveLinkedFeatures, saveGenes, saveLesions, saveGenotypingAssays, savePhenotypes, savePublications, addAssayFile, removeAssayFile, …) writes.
Pros Adding a new field is a one-line FIELDS map entry. Audit log is uniform — one row per change, regardless of which field. Section-scoped methods can validate the section atomically (e.g., "reasons[] and reasonsOther must be consistent"). Domain methods read more like business logic.
Cons The path-as-key indirection costs some readability vs. an explicit saveReasons(...). Field-path PATCH can't express "validate this whole section as a unit" — each path is validated in isolation. Per-section endpoints scale linearly with structured data types. Linked features, genes, lesions, phenotypes, publications each needed their own method. The service file is twice as long as ours and growing.

Net assessment

This branch is better at:

  • Long-run schema drift control — server-published JSON Schema beats anything in the alt branch.
  • Audit and observability — DB-backed audit + OpenAPI + drift test gives a queryable, contract-tested API.
  • Concurrency safety at the persistence layer — @DynamicUpdate is structurally stronger than a per-tab queue.
  • Library leverage — JSON Forms gives us conditional reveal almost for free.

ZFIN-10265 is better at:

  • Feature breadth — covers linked features, genes, lesions, phenotypes, autocomplete; we covered only the spine.
  • Per-field UX expressiveness — typeMatrices.ts carries help text, info links, placeholders, suffixes that are awkward to express in JSON Forms options.
  • Render-without-server — schema is local, so first paint doesn't wait on a fetch.
  • Existing test coverage — service-layer tests we don't have.

Honest verdict: if these were two PRs to evaluate side-by-side, the right answer is probably "take the alt branch's feature breadth and tests, replace its schema/render with our server-published JSON Forms approach, keep our audit table + OpenAPI + drift test." This branch's architectural bets are the right long-run direction; the alt branch's domain coverage is the right short-run answer. Treating them as competing rather than complementary is the false choice.

What went well, what we'd do differently

Wins

  • Backend as source of truth was the right call. This is our actual divergence from the alt branch — they kept schema authoring in TypeScript. Once we landed the server-published schema, every new section (3 in M3.3, 7 cluster-groups in M4.2) was almost no client work — just a new entry in FIELDS and the uiSchema group.
  • The FieldDescriptor pattern absorbed an "audit-capture switch" we never had to add. Audit, validation, and dispatch all share the same map.
  • Per-aggregate flat paths sidestep a real problem. JSON-pointer array-index resolution (/mutations/0/alleleDesignation) is solvable but adds a layer; we haven't needed it.
  • The seed-vs-autosave null gate is a one-line idiom that future-proofs any new aggregate against the same race.

Credit where due

Per-field PATCH, save-on-blur autosave, the save-status toast, the assay-type field-matrix idea, and the DTO shape we ended up keeping all existed first on ZFIN-10265. The full comparison is in the head-to-head section above; the short version is "both branches reached similar UX conclusions independently; our distinguishing bet is where the schema lives, not the UX it produces."

Regrets / things to revisit

  • We rolled our own diffLeaves, JSON-pointer paths, and stringify-key syncing when something like Immer + a real JSON Patch library could have given us cleaner primitives. We didn't reach for it because the leaf set is small (~25 paths per aggregate) and the diff is trivial to debug — but if we add 100+ paths the case gets stronger.
  • No DB-backed audit table yet. Audit log is log4j only. For curators it's behind an SSH session, not a UI. Eventually we want a zirc.audit table with the same row shape.
  • Renderers are hand-rolled. We chose this over JSON Forms' Bootstrap theme because the trunk reference markup is opinionated (table-row layout, ZFIN section style). It costs us per-renderer LOC.
  • The schema generators are imperative LinkedHashMaps, not records or builders. Readable but mistake-prone — e.g. forgetting to add a path to FIELDS that's referenced in the uiSchema only blows up at runtime when the user types in that field.
  • We didn't ship a section-level diff filter as a default. Each SchemaForm/MutationEdit/AssayEdit repeats the EXTERNALLY_MANAGED_PATHS idiom. A small shared autosave helper would deduplicate this.

Dev-practice debt worth paying down

Things that slowed us down across the milestones, in rough order of how much pain they caused:

Lombok-unaware Java LSP

The bundled LSP regularly reports "method undefined" against Lombok-generated getters. Touching an entity or a record that calls one via method reference fires 10–30 false-positive diagnostics. We learned to trust gradle compileJava instead, but the noise is constant and easy to mistake for real failures. Worth investigating a Lombok-aware LSP setup.

Tomcat OOM under long-running dirtydeploy sessions

After enough hot redeploys, the JVM heap fills up and we get OutOfMemoryError in every thread. A restart fixes it. Worth raising the dev-container's -Xmx or putting a watchdog on it.

Curl-poll after dirtydeploy

We were running until curl --max-time 5 ...; do sleep 5; done after every deploy to wait for Tomcat. That cost 30–60s per cycle. The next test step (Playwright navigate, curl probe) gives a clean "not ready" error if it's too early, so the polling loop was buying nothing. Cut it.

Working tree noise

The repo accumulates .png spikes, .zip dumps, and scratch SQL files in the working directory. git status on this branch is 60+ untracked entries deep. A .gitignore sweep would make commit-by-file workflows cheaper.

Liquibase tracking table out of sync on local dev DBs

The dev container's DATABASECHANGELOG table thinks several zirc changesets are still "pending" even though their SQL has been applied (it appears the original M3 schema went through raw SQL rather than Liquibase). gradle liquibasePostBuild therefore fails on dev — we ended up applying the M4.3 and audit-table migrations directly via psql for the local session, while the Liquibase changesets themselves are committed and correct for CI. A changelogSync on the affected rows would unstick this; worth a one-time cleanup.

Spock specs silently dormant in CI

See the OpenAPI section above. Capturing it here too because it has blast radius beyond the spec drift test — anyone writing a .groovy spec today is writing code that doesn't run. The fix needs to be opt-in (enable useJUnitPlatform() + add the vintage engine for existing JUnit 4 tests) and reviewed separately so a forest of dormant specs doesn't suddenly start failing as a side effect.

Long-running dev container OOMs under hot redeploy

Tomcat OOM'd three times during this session — each after enough dirtydeploy cycles. OutOfMemoryError in every thread, recoverable with a container restart. Worth raising the dev-container's -Xmx or adding a watchdog to restart on detected OOM. Doesn't matter in production where deploys happen fresh, but it's a real cost on the loop.

Memory system pays off

This is a small win, but worth flagging: the auto-memory entries for things like the seed-vs-autosave race, the Lombok LSP noise, and the curl-poll-after-deploy feedback survived a context-compaction and shaped the next session's decisions. The pattern of "save a short, actionable memory note when a non-obvious fix lands" is cheap to maintain and reliably prevents the same fights from re-running. The ~/.claude/projects/.../memory/ directory has a dozen of these now.

Publication Identification, Entry, Acquisition, and Annotation

This section was written/compiled by Ken in 2004/02.
Updated by Jonathan Knight 2006/06

The administrative coordinator conducts literature searches for papers about zebrafish periodically and enters the articles into ZFIN. Every night, a list of papers entered that day is automatically created and emailed to the curators. If there is any question about including a paper in ZFIN all curators should be asked for their opinion. If any curator thinks the paper should be included then the a publication record should be added to ZFIN. Papers that are "epub ahead of print" should be marked inactive and the paper should not be obtained until the publication date for that volume of the journal. The student assistant is responsible for acquiring these papers from online resources, the library, or inter library loan. Typically, if the paper does not contain any information currently annotated by ZFIN, there is no need to obtain a copy.. If there is any question if a paper should be obtained the point curator shoud be consulted.

If a paper is available as a PDF, the student assistant will attach it to the publication record in ZFIN. To attach a PDF file to the publication record, log into ZFIN and go to the publication record. Select the "Curate" hypertext link to the right of the ZFIN ID, select the "Browse" button, add the PDF file to the PDF box on the publication page, and select the "Upload" button. This will attach the PDF file to the publication page. If we have the journal's permission to use the images in the paper the images and associated figure legends should be uploaded at this time. The list of journals who have responded to our request for figure permissions is available from the ZFIN Journal Permissions Table

To add images to publications click on the "FX curation" link to bring up the FX annotation/update page.

Before adding a figure image, you first need to make a figure record in the database. In the Figure section of the FX form, just enter the figure number in the entry box next to Fig. in the Figure section. You don't need to enter "F-i-g", just enter the number. Then click the Add button to the left. Ta-da! You've made a figure record.

Images that should be uploaded are images of fish. Images that just contain diagrams or sequences should not be added.

To add an image to a figure, first download the image to your computer from the journal website (or extract the image from the pdf, if the publication is not available in HTML form on the journal website). Make sure that the image you download is the maximum resolution image available on the website - sometimes there is a thumbnail, then a medium sized image that you can click to get the full-sized image. Bigger is better.

In the Image section of the FX form, select the figure number from the Fig pull-down menu, then hit the Browse button to find the image on your hard-drive. Click 'Add' to upload the selected figure. You won't need to enter anything in the Label field unless (very rare) there are multiple images for one figure - in that case you would need to enter a Label like 'A' or 'B' to uniquely specify each image associated with the same figure.

You can enter the caption at the same time you create the figure record or later on. The display-version of the caption (what the users will see) is shown above the caption entry box - just click the 'Update' button to see the formatted results of your latest changes to the caption.

It's best to enter plain text with explicit HTML markup, and then to carefully check the ZFIN display version with the journal caption. You can set your browser to display the HTML source for caption-making. The most common markup types are HTML italics for gene and mutant names, and the Greek letter mu, for micrometers in the scale bars.

A hard copy will also be printed for curation, labeled with the ZDB-PUB ID number and placed in the "To Be Curated" basket in the main office area. If images have been uploaded an indication of this should be made on the printed copy. Currently a star is drawn by the ZDB-PUB ID number. When you are ready to curate a paper, take a paper out of this bin and capture the information as outlined below.

When you have completed extracting the information from the paper and entered it into ZFIN, go to the publication record, and select the "Curate" hypertext link. There you will find a set of data types with a series of boxes. Select the boxes on the far left next to the data types found in the paper. The categories are: Expression, GO, Genes, Mapping, Mutant, Nomenclature, Orthology, Phenotype, and Sequence. Most of these categories are intuitive except the distinction between "Genes" and "Nomenclature". Select the "Genes" box if there is a new gene in the paper. The "Nomenclature" box is reserved for cases where there was information in the paper that caused you to change the nomenclature of an existing gene.

For each data type you have selected on the publication page, if you have annotated that information to completion, select the corresponding "Close" box to the right of the data type. If you have not annotated that information, select the corresponding "Open" box. By default, you will be selected as the "Curator" when you select the "Open" box. If you want to change the curator in that field, you can by using the pull down menu. Instances in which you might do this is if the data are complicated or presented in a confusing manner and you want to hand the paper off to someone with more expertise in annotating that particular type of data. Make sure you notify a person that you have selected them to curate information in a paper. There is also a non-public "Notes" field if you would like to make any comments. If you have annotated all of the information in a paper, select the "Close Paper" button to indicate the paper is fully annotated.

When you are finished with the paper, file put it in the "To be Filed" basket in the main office area.. If you need to check out one of these papers, fill out a card and include the ZDB-PUB ID, your initials, and the date on which you took the paper. This card should be placed in the location where the paper should be filed. The cards allow someone to know who has that paper if it is missing, in addition to allowing you to file the paper more quickly and accurately when you are finished with it Be sure to file it properly so that it can be easily found at a later date.

Zebrafish Anatomical Ontology

Editing the Zebrafish Anatomical Ontology (AO) in DAG-Edit

Point person currently Melissa Haendel
Backup person Ceri Van Slyke

The zebrafish anatomy ontology (AO) is currently edited in DAG-Edit version 1.419 rev3. This will change when the new OBO-Edit is available in full release. The program is available for download at: http://sourceforge.net/project/showfiles.php?group_id=36855

The following fields are utilized in DAG-Edit:

  • ID: new anatomy terms are assigned a unique ID of the following form: ZFA:#######. ZFIN stages, which are constant, have the form ZFS:#######. These IDs can be set by each editor in the DAG-Edit configuration Manager.

  • Namespace: there are two namespaces for anatomical terms and stages, zebrafish_anatomy and zebrafish_stages, respectively. All are found under the umbrella namespace zebrafish_anatomical_ontology.

  • Term name: some guidelines for adding new terms are 1) Plurals are used only when there are children of the term (or will be). Exceptions occur when putative children have no specific names. For example, scales. 2) No capital letters except when it is a name. 3) No left/right designations unless the terms differ on the two sides.

  • Definition: Definitions may come from a collaborator or from the literature and may be linked to pubs or websites using html sourcecode.

  • Synonyms: Synonyms may come from a collaborator, the literature, or be a previous name for a term. These should include plural/singular variations.

  • Dbxrefs: a ZFIN ZDB-ANAT ID is displayed. These come from the database once the AO is loaded and a new obo file generated. In the case of adding a cell type from the CL ontology (see below), the CL identifier would be added here.

  • Parent Plugin: For every term added, a start_stage and end_stage are added, as well as a relationship to the term’s parent. These relationships are is_, part_of, or develops_from as defined by OBO. Terms must follow the rules that for is_a and part_of relationships, a child must exist within the range of its parent. For develops_from relationships, terms must overlap or abut in time.

  • Comments: It is suggested that the comments field be utilized for explanations of new term creation, term relationship changes and any additional notes such as which collaborator has approved a definition.

  • Currently, terms from the cell type ontology can be added to the AO. However, they will keep their CL identifiers such that when the cell type ontology is implemented independently they can be separated. In order to do this, add the cell type with a ZFC:####### ID and put the CL:####### in the Dbxrefs field.

  • Merging terms: Do not create a new term and then merge it into another term. The reverse, merging an existing term into a new term, is ok. Because the direction of the merge decides the order of the two or more ZDB-ANAT- id lines, and further decides which id gets kept and which gets tossed, be conscientious of the direction. For example, if A is merged into B, and really should be B merged into A, then undo the merge and re-merge the other way around.

  • Curators may view the history of changes to a term by clicking on the “Updated: date” button on the upper right corner of an anatomy page.

  • Loads of the AO into the database should occur every month or two, dependent on how many changes have been put into the CVS.

Using CVS with the AO

CVS stands for concurrent version system. This means a file can be edited by more then one person at a time. In order to be able to get a copy of the file from CVS it is necessary to have a CS account, secure shell client (SSC) and secure file transfer client (SFTC) software, and have someone with root access make you part of the group with permission to edit the file in CVS.

Each time a curator works on the file they should get the most up to date version of anatomy.obo. At the end of each work day their edited file should be committed to the repository.

The first time you get the anatomy.obo file from CVS, after logging in using the SSC you will need to type "cvs checkout Curation". This will give you access to the anatomy.obo file.

After that, each day before you start your work, login in using SSC and do a "cvs -nq update"(if you want to be very very careful) and "cvs -q update" to bring in the current version in the repository to your working directory. If you see a "U" in front of the file, you know somebody else updated the file after your last commit. This file then should be copied to your desktop using SFTC to be updated using DAG/OBO-Edit.

After editing, the file should be renamed and SFTC should be used to copy the file back to your directory and the new file copied over the old copy of anatomy.obo. Now the anatomy.obo file in your directory has your day's work, you do a "cvs -q update" to bring/merge in the current version in the repository. You will see either "M anatomy.obo" or "C anatomy.obo". "M" stands for your file has modifications (no matter whether there is updates in the repository or not). "C" stands for conflicts during the merge, which tells you there are updates in the repository and that the updates somehow conflict with your modifications. In this case, you need to edit anatomy.obo file to resolve the conflicts. Conflicts are in blocks marked by

   <<<<< r.#
     content in this repository version
   = = = =
     content in your working version
    >>>>>>

Only after the conflicts are resolved, would you be able to commit your working version into the repository.

To commit the file to the repository you type "cvs commit -m "comments" anatomy.obo". The comments should be inside quotes and should briefly note the type of changes you made. CVS will give you a new version number.

The most common commands used by the curators are:
cp filename1 filename2 -- copies file1 to file2 keeping a copy of file1
cvs -nq update -- Show if files have changed but doesn't do anything to files
cvs -q update --Bring updates into your copy of the file
cvs commit -m "comments on updated file" anatomy.obo -- commits changes to CVS with your comments
cvs diff -rversion number1 -r version number2 anatomy.obo |less -- shows a list of changes between versions of the two files. The version numbers should be in the form 1.21. This list uses greater or less than signs to show differences between the files. The |less lets you see one screen at a time and advances the view through the list when you hit the space bar. To exit type q.

If you want to minimize the need of using a text editor to reconcile differences in the file have only one curator at a time work on the file.

General commands for use with CVS are available in the doc at http://mirror.zfin.org/zfin_doc/impl.html#Source%20Code%20Control%20and%20CVS
Most of these commands are not routinely used by the curators that edit the AO file.

Loading changes to the AO into the database

Point person currently Peiran Song
Backup person Sierra Taylor

Where Are The Scripts

Scripts for AO loading and unloading are at ZFIN_WWW/server_apps/data_transfer/Anatomy. There is a README file documenting the steps to conduct AO load and unload with brief explanation of each script. For data load and unload purpose, please read and follow the README file.

The following documentation goes into details on what things are done and how they are done. It is for debugging and revision purpose.

Input File Format

AO Term:
Below is a typical AO term in OBO format. Some notes:
1) Brand new terms do not have xref_analog line.
2) When terms merge, the old(merged) term’s name becomes the new(merged-into) term's synonym, ID becomes an alt_id: ZFA:# line, and if exists xref_analog, it would be inserted in front of the new term’s xref_analog line.
3) When terms become obsolete, there will be an is_obsolete: true line.

[Term]
id: ZFA:0000006
name: ball
namespace: zebrafish_anatomy
xref_analog: ZFIN:ZDB-ANAT-010921-413
synonym: "yolk ball" [ZFIN:ZDB-PUB-961014-576]
def: "The anterior round region of the yolk cell present after the yolk extension forms during the segmentation period. <a href='http://zfin.org/cgi-bin/ZFIN_jump?record=ZDB-PUB-961014-576'>Kimmel et al, 1995.</a>" [ZFIN:curator]
relationship: start ZFS:0000025
relationship: end ZFS:0000036
relationship: part_of ZFA:0000084

Cell Ontology term:
We have also incorporated terms from the Cell Ontology. Below is a typical new CL term in OBO format.

[Term]
id: ZFC:0001318
name: lymphopoietic stem cell
namespace: zebrafish_cells
is_a: ZFC:0001317 ! hematopoietic stem cell
relationship: end ZFS:0000044 ! Adult
relationship: start ZFS:0000030 ! Pharyngula:Prim-15
xref_unknown: CL:0000051

What Happens During The Loading

1) Parser generates

  • one translation file:
      cell_ids.unl : ZFC:#######|CL:#######|
  • anatitem_new.unl : terms without xref_analog line, one term per line
  • anatitem_exist.unl : terms with xref_analog line, one term per line
  • anatitem_merged.unl : ZFA:#######|ZFA:#######|
  • anatitem_obsolete.unl : ZDB-ANAT-######-##|
  • anatalias.unl : ZFA:#######|synonym|ZDB-PUB-######-##|
  • anatrel.unl : ZFA:#######(parent)|ZFA:#######(child)|rel_type|

2) Loader has two parts, the first part creates temporary tables, load raw data in, use the translation file, anatomy_item and stage table to translate the ZFA/ZFS/ZFC ids to ZDB ids. It also executes create_anatomy_relationship_list() procedure to generate parents and children lists for each anatomy term, store them for later comparison.

The second half loads data into real tables as explained below:

For existing terms: update it in anatomy_item, record actions in updates table.
For new terms: enter it to zdb_active_data and anatomy_item
For merged terms:

  1. expression annotations and pattern figures:
    If same annotations exist on both old(merged) and new(merged-into) terms, keep the ones on the new terms and link xpat figures from the annotation on the old term over to the annotation on the new term. For the rest annotations on the old terms, do a distinctive select, and generate new -XPATRES- ids (it is possible that two old terms with the same annotation get merged into the same term, we choose to get a new id instead of picking an old one out.) Then link over all the xpat figure records on the old annotations to the news. For deletions on expression annotation, either due to both new term and old term have the same annotation or more than one terms with the same annotation merged into one term, record annotation ids in zdb_replaced_data.
  2. anatomy items:
    Record term zdb ids in zdb_replaced_data, record merging in updates table for the new terms, delete the old terms from zdb_active_data. The term is gone but the term name is kept as the new term's synonym.

For obsolete terms: flag it in the anatomy_item
For synonym/alias updates:

  1. record synonym deletions in updates table for synonymns that are not included in the new AO file, then delete those synonyms from zdb_active_data.
  2. for existing synonymn attributions, delete those that are not included in the new AO file, and add those they are new.
  3. add new synonyms to zdb_active_data and data_alias, and add attribution to record_attribution.

For Cell ontology term: use CL:# instead of ZFC:# as anatitem_obo_id.
For relationship: this is where stage inconsistency often comes up.

  1. three checkings correspond to three .err output files:
    start_startInconsistent.err: parent term|child term|parent start stg|child start stg|
    end_startInconsistent.err: parent term|child term|parent end stg|child start stg| (for develops_from relationship)
    end_endInconsistent.err: parent term|child term|parent end stg|child end stg|
  2. load in anatomy_relationship
  3. run create_anatomy_relationship_list() procedure, record change to each term's contained_by, contains, develops_from and develops_into lists.

To have history tracking, we store each update/change action into the updates table, and the web application function would retrieve and format them for web display.

Updates that we track include:

  • term name change
  • term start stage or end stage change
  • term definition change
  • term receives a merge
  • new synonym
  • contained_by (is_a, part_of) parent list change
  • contains (is_a, part_of) children list change
  • develops_from parent list change
  • develops_into children list change

3) With term stage(s) changes, some existing expression annotations might have stage inconsistence problem. The loader generates annotationViolates.err file in a format of:
ZDB-PUB-###(annotation source)|annotation start stage abbrev|end stage abbrev|anatomy term name|term start stage abbrev|end stage abbrev|

Curators would return a file in format of
ZDB-ANAT-####(term to be translated)|start stg abbrev|end stage abbrev|ZDB-ANAT-###(term to be translated into)

These file would be used by batch_xpat_annot_adjust() function to adjust annotation.

4) Providing Thisse keyword file--stageKeywords.csv, loadMaster.sh would parse it and check it against new AO table content and generate two files to curators:

kwdNameNotPrim.err: start stage name|end stage name|term name|
The term name used in that stage window is not longer a primary ZFIN AO term name, it might become a synonym, or updated or merged or obsoleted.

kwdStageInconsis.err: start stage name|end stage name|term name|
The term name used in that stage window is not longer consistent with current stage definition of that term in ZFIN.

What To Do After The Loading

First, commit the anatomy.obo file that used for the loading into CVS.

Then, unload ZFIN anatomy and stages.
During the unload, each anatomy term grabs its parents from anatomy_relationship, its synonyms from data_alias and synonym attribution from record_attribution, and form an obo format item definition.

We generate two version of obo file,

  • one with term comments(anatitem_description) and cell ontology terms for internal use

  • one without both for sharing at public OBO site

Both files would be sent to curators via emails as the end of the execution. Curator(s) would re-save them by DAGEdit for format/order adjustment. The internal one should be committed into CVS by the curator.

Finally, the AO files at ZFIN public download site would be updated on Sunday night by weekly cron job, or you could choose to launch it right away.

Making changes to anatomy PATO templates

Point person Erik Segerdell
Backup person Ceri Van Slyke

A FileMaker Pro 7 database (or "template") has been developed for phenotype annotation based on the syntax and ontologies now being devised by the Phenotype Ontology consortium.

Unlike the case of the Thisse expression pattern template, we do not receive routine data submissions via the PATO template. It was developed originally for trial curation of mutant phenotypes based on the PATO schema. We also have distributed copies to, among a few others, Makoto Furutani-Seiki (medaka fish mutant screen) and to Paula Mabee (integrated fish evolution database) for their own particular projects. See FogBugz case 587 for more information.

The PATO template uses several controlled vocabularies for annotation. When AO updates occur, this template should be kept in sync, and the above users should be notified.

Updating Anatomical Terms

The template consists of four integrated files; approved anatomical terms are stored in rel_pato_val_anat(.fp7). No experimental data is ever stored in this file. It simply provides a value list of terms that shows up in an Anatomy pick-list during annotation. Once it is updated, users need only to replace this particular file on their machines.

Unlike the Thisses, who use a preferred subset of AO terms for their annotations, the PATO template stores all AO terms and their ID's. This makes updating the file particularly simple:

  • Obtain a flat file containing ID's and anatomy items that reflect the most recent AO update and save it on your machine:
    • http://zfin.org/data_transfer/Downloads/anatomy_item.txt

  • Open only rel_pato_val_anat(.fp7). In FileMaker, select Records -> Show All Records, then select Delete Found Records...

  • Select File -> Import Records -> File...

  • In the dialog box, open the saved flat file.

  • In the next dialog box, you should see ZDB-ANAT-ID's mapped to the anat_id target field, and the anatomical terms themselves mapped to the anat_term field. If not, adjust the target fields. ZDB-STAGE-ID's will not be imported.

  • Click Import to complete the procedure.

Genes

This section was written/compiled by Ken in 2004/02.

Gene Nomenclature

One of the first things you'll need to do is figure out the gene(s) associated with the data in the paper. They might be existing genes, completely novel, or novel to zebrafish, but known in another species. It is important for curators to do independent analysis of genes reported in papers and not take the word of the author that it is either novel or an existing gene.

There are a number of online resources, analysis tools, and documents instrumental in researching the identity of the gene and, if necessary, assigning nomenclature. They are too numerous to list, but can be found in the curation resources section. It is to your advantage to learn about and become comfortable with these online resources, analysis tools, and documents. Information in those resources will not be repeated here to any great extent.

If the authors report nucleotide sequence accession numbers associated with a gene, search ZFIN using the number to check to see if the number is already in ZFIN associated with a reserved gene or a gene record created for a GenBank number. If nothing is returned, run a blastn and blastx against the nonredundant (nr) database with the reported sequence to investigate whether it matches a published or unpublished sequence and whether that sequence is already in ZFIN associated with a gene record.

If a BLAST analysis returns a matching sequence that is listed as unpublished, it doesn't mean that it is unpublished. Select a couple authors from the "unpublished" GenBank record and type them into PubMed to look for a published paper containing this sequence. If you find a matching published zebrafish sequence in an unannotated paper, acquire the paper and annotated it at the same time you are annotating your original paper.

If there is no nucleotide sequence accession number presented in the paper, sometimes there is a nucleotide or protein sequence presented in a figure Typing in a portion of the sequence and running a BLAST analysis may identify a longer or even complete zebrafish, mouse or human sequence which can then be used for more comprehensive analysis. If the sequence in the figure is long, it is wise to type in portions of the 5', central, and 3' portion of the sequence to increase the chances of finding a partial gene sequence deposited in GenBank.

Let's assume your initial investigations suggest that the gene you are analyzing is novel. It is important to run a BLAST analysis against the zebrafish ESTs to identify genes in ZFIN that are only known as ESTs. If a published zebrafish sequence matches an EST that is associated with a gene whose nomenclature is based on EST nomenclature it is important to analyze the published sequence and rename the gene with more informative nomenclature.

From your initial sequence similarity analyses, you should get a feel for the relationship of the gene under consideration to other zebrafish, mouse and human genes. For example, you may be able to tell that it is a member of an established gene family based on similar identity to other zebrafish sequences and that is the likely ortholog of a particular human or mouse sequence. It is important to use this information in conjunction with the information provided in the paper such as author's statements, sequence alignments, phylogenetic trees, and experimental data such as expression patterns.

Elucidating relationships may be furthered by looking for conserved synteny using analysis tools at Ensembl. BLASTing the sequence under consideration against the zebrafish sequences at Sanger may identify a contig on which the gene is found. Looking at the flanking zebrafish genes on that contig, their likely human or mouse counterparts based on Sanger's computations, and the neighborhoods surrounding those human and mouse genes, may provide evidence of orthology based on conserved synteny.

If the gene under consideration is novel to zebrafish and is the ortholog of a human or mouse gene, naming is pretty straight forward as gene nomenclature follows that of human. If there is no human ortholog, the nomenclature follows that of mouse. Often human gene names will contain terms or descriptors in brackets which seem excessive. These are not included in the zebrafish name. It is a judgment call whether to include the terms or not. In most cases, these extra descriptors are easy to pick out. I usually look at the gene names of zebrafish genes in the same gene family if there are any. Another method to identify these terms in the human gene name is to look at the gene name of the orthologous mouse gene and see if they are present.

If two zebrafish genes are determined to be the genome duplicates of a human or mouse ortholog, the mammalian gene and name nomenclature is used, followed by an 'a' and 'b'. Typically, duplicate nomenclature is not used unless there is evidence that the two zebrafish genes are duplicates. Closely related zebrafish genes with high sequence similarity should not be given duplicate nomenclature based on sequence alone. Corroborating evidence that they are genome duplicates can be conserved location of the genes with other known zebrafish genome duplicates, conserved location with human or mouse orthologs, and analyses using phylogenetic software programs such as clustal.

When two genes are determined to be genome duplicates, base your assignment of 'a' and 'b' on the nomenclature of other genome duplicates on the same chromosome. For example, if you identify genome duplicates on chromosome 3 and 12, search ZFIN for genes on those two chromosomes and see whether the named 'a' duplicates are on 3 or 12. In this example, the hoxb#a genes are on 3 and the hoxb#b genes are on 12. The duplicate on 3 would be 'a' and duplicate on 12 would be 'b'. It is impossible to be completely consistent in naming, but try where you can.

If there is no human or mouse ortholog, try to identify if it is a member of a gene family and use the gene family nomenclature. If the zebrafish gene is a novel member of a gene family, but its relationship to human or mouse gene family members is unclear, give the zebrafish gene the gene family root symbol and use a number if the human and mouse nomenclature use letters and vice versa. For example, there are a number of claudin genes in zebrafish (symbol: cldn*), some of which are identified as orthologs of human and mouse genes while others have undetermined relationships with mammalian counterparts. The claudin genes with established human and/or mouse orthologs are given the same nomenclature. An example of this is cldn10 whose human ortholog is CLDN10 and mouse ortholog is Cldn10. A zebrafish claudin gene whose relationship to human and mouse claudin genes is undetermined at this point is cldnf. Once the relationship of this zebrafish gene is clarified in relation to the human and mouse claudin genes, cldnf will be renamed using a number corresponding to the orthologous gene in human and mouse.

If it is not in a gene family, look for domains or conserved regions using Pfam and try to utilize that in the name. On occasion, there will be information in the GenBank record helpful in proposing a name, however DO NOT TRUST THE NOMENCLATURE IN GENBANK. It is very unreliable.

It is OK to use what the authors use if it resembles approved zebrafish nomenclature format, there is no better alternative, and it does not conflict with existing gene or mutant nomenclature in zebrafish, human, or mouse. It is important to make sure the proposed nomenclature does not interrupt or become easily confused with that of existing zebrafish, human, or mouse nomenclature.

"Reason"s for renaming gene names and symbols

When renaming genes, there is an opportunity to indicate a "reason". The options and instances for which you use each are:

"per gene family revision" - If the gene family undergoes revision and names change because of the work done.

"per personal communication with authors" - communication with an author leads to a change in the nomenclature.

"renamed to conform with human guidelines" - the nomenclature for the human ortholog changed from one approved symbol to another approved symbol. The nomenclature of the zebrafish ortholog is now out of sync and is being changed. Another scenario in which to choose this option is when you are renaming uninformative zebrafish gene nomenclature with unapproved human gene nomenclature found in LocusLink.

"renamed to conform with mouse guidelines" - there is no human ortholog and the nomenclature for the mouse ortholog changed from one approved symbol to another approved symbol. The nomenclature of the zebrafish ortholog is now out of sync and is being changed.

"renamed to conform with zebrafish guidelines" - use this option for most naming not covered by other options. This includes if the zebrafish gene is now found to be the ortholog of a human gene with different nomenclature. Another example would be if an existing zebrafish gene is found to be a genome duplicate and the duplicate nomenclature conventions are being instituted.

"not specified" - probably used only in very strange scenarios, if ever.

"same marker" - automatically listed when you merge a gene. Should you delete a gene and realize you should have merged it, this is the reason you use. It will probably never be used unless and error is made and you have to clean up.

General GO Consortium Recommendations

(modified from www.geneontology.org/GO.annotation.shtml)

  • Annotate gene products to the most detailed level in the ontology that correctly describes the documented biology of the gene product. A gene product can be annotated to zero or more nodes of each ontology.
  • Keep the True Path Rule in mind: annotating to a term implies annotation to all parents via any path. So all the parent terms must be true as well. It is a good idea to check the parentage of a term before annotating (and request new terms or path corrections if necessary).
  • There is an important distinction between a gene/gene product annotated to "unknown" function, process, and/or component, and one that has not been annotated. "Unknown" means that someone has tried annotating the gene, but didn't find any information. Absence of annotation implies that no one has looked. Curators are encouraged to annotate to terms from all three ontologies, using "unknown" liberally if necessary. For annotations to "unknown" from any of the three ontologies, you should cite our pseudopub which explains the application of this GO term. The correct pub and evidence code are populated in the corresponding fields for you when you select one of the three "unknown" GO terms.
  • Uncertain knowledge of where a gene product operates should be denoted by annotating it to multiple terms, one of which can be a parent of the other. For instance, a gene product known to be in the nucleolus, but also experimentally observed in the nucleus generally, can be annotated to both nucleolus and nucleus in the cell component ontology. Even though annotation to nucleolus alone implies that a gene product is also in the nucleus, annotate to both so as to explicitly indicate that it has been reported in the two locations. The two annotations may have the same or different supporting evidence. Similar reports of general and specific molecular function or biological process for a gene product could be handled the same way. You also can annotate to multiple nodes that conflict with each other if there are conflicting claims in the literature.
  • A gene product should be annotated with terms reflecting its normal activity and location. A function, process, or localization (component) observed only in a mutant or disease state is therefore not usually included.
  • An individual gene product that is part of a complex can be annotated to terms that describe the action (function or process) of the complex. This is colloquially known as annotating 'to the potential of the complex', and is a way to capture information about what a complex does in the absence of database objects and identifiers representing complexes. It is especially useful in cases where a complex has an activity, but the individual subunits do not. One example is the subunits of nuclear RNA polymerases, where none of the individual subunits have RNA polymerase activity, yet all of these subunits are annotated to 'DNA-dependent RNA polymerase activity', to capture the activity of the complex. In such cases you should use the "contributes_to" qualifier to indicate that a gene product contributes to the function of a complex, but does not have, or it is not known to have the function itself.

IC - inferred by curator

  • GO terms implied by the presence of other GO terms with non-ISS/non-IEA evidence

To be used for those cases where an annotation is not supported by any evidence, but can be reasonably inferred by a curator from other GO annotations, for which evidence is available. Evidence is defined as an evidence code other than ISS or IEA. Use TAS or NAS rather than IC if there are author statements to support the derived GO term.

For example if there are experimentally supported GO annotations indicating that a particular gene product has the function "transcription factor activity", but there is no evidence whatsoever that this gene product has the cellular location "nucleus", this would be a perfectly reasonable inference to make.

The inferred annotation "nucleus" should be linked to the "transcription factor" annotation in two ways:

  • The derived annotation (nucleus) and source annotation (transcription factor activity) should cite the same publication. If the derived term uses a source GO term attributed to multiple publications, it is OK to cite only one of those papers in the derived annotation.
  • The "inferred from" field of the derived annotation should include the GO ID of the source GO term (GO:#######).
[IC documentation at GO website]

IDA - inferred from direct assay

  • Enzyme assays
  • In vitro reconstitution (e.g. transcription)
  • Immunofluorescence (for cellular component)
  • Cell fractionation (for cellular component)
  • Physical interaction/binding assay (sometimes appropriate for cellular component or molecular function)

This evidence code is used to indicate a direct assay for the function, process, or component indicated by the GO term. Curators therefore need to be careful, because an experiment considered a direct assay for a term from one ontology may be a different kind of evidence for the other ontologies. In particular, direct assays for cellular component are more common than for function or process.
For example, a fractionation experiment might provide IDA evidence that a gene product is in the nucleus, but IPI evidence for its function or process. Binding assays can provide direct assay evidence for "... binding" molecular function terms.

[IDA documentation at GO website]

IEA - inferred from electronic annotation

  • Annotations based on "hits" in sequence similarity searches, if they have not been reviewed by curators (curator-reviewed hits get ISS)
  • Annotations transferred from another database's records, if not reviewed by curators. An example of this is applying GO terms to genes through the use of the spkw2go translation table.

This evidence code is used for annotations that depend directly on computation or automated transfer of annotations from a database. The key feature that distinguishes this evidence code from others is what a curator has done--IEA is used when no curator has checked the annotation to verify its accuracy. The actual method used (BLAST search, SwissProt keyword mapping, etc.) doesn't matter.

[IEA documentation at GO website]

IEP - inferred from expression pattern

  • Transcript levels (e.g. Northerns, microarray data)
  • Protein levels (e.g. Western blots, immunohistochemistry)

Covers cases where the annotation is inferred from the timing or location of gene expression. Expression data will be most useful for Process annotations. For example, several of the heat shock proteins are thought to be involved in the process of stress response because they are upregulated during stress conditions. Another example might be in microarray data, where a set of known genes respond to an ongoing process, and a set of unknown genes show a similar reponse. This expression pattern suggests that the novel genes are involved in the process. Generally, a publication must demonstrate a correlation between the expression pattern of a gene and a known process before this evidence code can be used. Use this code with caution!

[IEP documentation at GO website]

IGI - inferred from genetic interaction

  • "Traditional" genetic interactions such as suppressors, synthetic lethals, etc.
  • Functional complementation
  • Rescue experiments
  • Inference about one gene drawn from the phenotype of a mutation in a different gene

Includes any combination of alterations in the sequence, expression, or function of more than one gene/gene product. IGI is appropriate when experiments are done with fish harboring mutations in two or more genes, or when morpholinos or overexpression is used with more than one gene at a time. When redundant copies of a gene must all be mutated to see an informative phenotype, that's IGI.

An entry in the "inferred from" field is REQUIRED when using this evidence code. This "inferred from" entry should be an identifier for the other gene(s) involved in the interaction. If mutant fish are used, the ZDB-FISH ID (allele) should be used. If the exact allele is not known, use the ZDB-FISH (allele) ID for the "not specified" allele. If morpholinos are used to reduce expression of one of the participating genes, use the ZDB-MRPHLNO ID. If multiple independent interactions between pairs of proteins support the same GO term (i.e. mutation of A and B, and mutation of A and C), separate annotations should be created for each independent piece of "inferred from" data. In cases where interaction between more than two proteins is required (i.e. mutation of genes A, B and C is simultaneously required), all of the interacting genes should be grouped together into one annotation.

To help clarify:

GOterm    IGI    ZFIN:gene1,ZFIN:gene2
means that the GO term is supported by evidence involving simultaneous interaction with *both* of these genes;
i.e. neither of the following statements are known to be true:
GOterm    IGI    ZFIN:gene1
GOterm    IGI    ZFIN:gene2

The "inferred from" field can be used in three ways with IGI evidence.

  • To indicate a single partner in the interaction enter a single item in the "inferred from" field.
  • To indicate that an interaction with more than 1 protein is required for the annotation to apply, enter multiple comma-separated items in the "inferred from" field. These will be displayed grouped together.
  • To indicate that multiple independent interactions support a single GO term, make a new GO annotation for each piece of evidence.
[IGI documentation at GO website]

IPI - inferred from physical interaction

  • 2-hybrid interactions
  • Co-purification
  • Co-immunoprecipitation
  • Ion/protein binding experiments

Covers physical interactions between the gene product of interest and another molecule (or ion, or complex). For functions such as protein binding or nucleic acid binding, a binding assay is simultaneously IPI and IDA; IDA is preferred because the assay directly detects the binding.

An entry in the "inferred from" field is REQUIRED when using this evidence code. This should be the UniProt identifier for the protein involved in the interaction. Often the authors won't report the UniProt ID of the protein, but you should be able to use the generic UniProt ID for the gene. If multiple independent physical interactions support the same GO term, separate annotations should be created for each independent piece of "inferred from" data. In cases where the protein of interest interacts simultaneously with more than one other protein, all of the interacting proteins should be displayed grouped together.

To help clarify:

GOterm    IPI    SPTR:protein1,SPTR:protein2
means that the GO term is supported by its interaction with *both* of these proteins;
i.e. neither of these statements are known to be true:
GOterm    IPI    SPTR:protein1
GOterm    IPI    SPTR:protein2

As with IGI, the "inferred from" field can be used in three ways with IPI evidence.

  • To indicate a single partner in the interaction, enter a single item in the "inferred from" field.
  • To indicate that an interaction with more than 1 protein is required for the annotation to apply, enter multiple comma-separated items in the "inferred from" field. These will be displayed grouped together on one line.
  • To indicate that multiple independent interactions support a single GO term, make a new GO annotation for each piece of evidence.
[IPI documentation at GO website]

IMP - inferred from mutant phenotype

  • Any single gene mutation/knockout
  • Overexpression/ectopic expression of wild-type or mutant genes
  • Anti-sense experiments and morpholinos
  • RNAi experiments
  • Specific protein inhibitors

We will use this frequently when mutant fish, morpholinos, or transgenic fish are used. Anything that is concluded from looking at mutations or abnormal levels of the product(s) only of the gene of interest is IMP (compare IGIs where multiple genes/gene products are involved). Use this code for experiments that use antibodies or other specific inhibitors of RNA or protein activity, even though no gene may be mutated. The rationale is that IMP is used where an abnormal situation traceable to a single gene prevails in a cell or organism.
If your annotation is derived from studies using mutant fish, put the ZDB-FISH (allele) ID in the "with" field. If the exact allele is not known, use the ZDB-FISH (allele) for the "not specified" allele.
If your annotation is derived from studies using morpholinos or other types of inhibitors, please enter the ZDB-ID of the inhibitor in the "with" field when such an ID is available. Use the notes field liberally to add detail to the annotation...we may draw on this information later.

[IMP documentation at GO website]

ISS - inferred from sequence or structural similarity

  • Sequence similarity (homologue of/most closely related to)
  • Recognized domains
  • Structural similarity
  • Southern blotting

Use this code for sequence similarity results (BLAST, InterPro, etc) that have been reviewed for accuracy by a curator. ISS can also be used for sequence similarities reported in published papers, if you think the result is reliable enough.
If the same annotation can be made from a single paper based on sequence similarity with multiple sequences then multiple annotations should be made with a single sequence used in the inference data for each (ie..one annotation for similarity to mouse and one annotation to for similarity to human). Rare exceptions may occur when grouping the inferences together would be appropriate...for example when sequence similarity with two interpro domains suggests a function (nuclear localization signal and zinc finger domains might support ISS for transcription factor activity)

An entry in the "inferred from" column is required when using this evidence code. This should be the identifier for the similar sequence, and will generally be a SwissProt ID, RefSeq ID, or InterPro ID.

Getting Annotations From an Ortholog
If there is no GO in a category and no annotations can be obtained from publications, you can adopt annotations from the mouse or human ortholog using ISS under the following circumstances:
A) The sequence similarity between the ortholog and the zebrafish gene is >50% (this is an arbitrary cutoff, but we need to have one...do a BLAST analysis)
..AND..
B) The term to be adopted is supported by experimental evidence on the ortholog (ie. IDA, IMP, IGI or IPI).

This is generally a viable option for Molecular Function terms. However, Process and Component annotations should be brought forward from another species with caution, as these can be split among gene duplicates in zebrafish. The "inferred from" field should contain either the nucleotide or protein RefSeq ID (NM_ or NP_ respectively) taken from the ortholog's gene record. Cite the ZFIN pseudo-publication that has been established specifically for this type of annotation (ZDB-PUB-040216-1).

NAS - Non-traceable author statement

  • Statements in papers (abstract, introduction, or discussion) that a curator cannot trace to another publication

This one seems pretty self-explanatory...

ND - no biological data available

  • Used for annotations to "unknown" Molecular Function, Biological Process, or Cellular Component after a curator has determined that no information is available for the marker.

This code is used only for annotations to "unknown," and conversely it is the only evidence code that should be used for annotations to unknown. When applying this annotation, use the ZFIN pseudo-pub that has been specifically established to support this annotation. If you select a GO "unknown" term, this pub and the ND evidence code will automatically be populated in the curator interface for you. TAS - traceable author statement

  • Anything in a review article where the original experiments are traceable through that article (material from introductions to non-review papers will sometimes meet this standard)
  • Anything found in a text book or dictionary; usually text book material has become common knowledge (e.g. "everybody" knows that enolase is a glycolytic enzyme).

TAS and NAS are both used for cases where the publication that a curator uses to support an annotation doesn't show the evidence (experimental results, sequence comparison, etc.). TAS is meant for the more reliable cases, such as reviews (presumably written by experts) or material sufficiently well established to appear in a text book. There isn't really a sharp cutoff between TAS and NAS. Rather than use TAS to make annotations from a publication, You should try to get the source publication when possible if an author states something like "The doug1 gene encodes a bHLH transcription factor known to be involved in gastrulation(Howe et al., 1999)".

ZFIN GO curation Policies:

Annotating a subunit of a complex and the use of the "contributes to" checkbox.

  1. Annotate the individual subunits of a complex with the Cellular Component term for the complex.
  2. If the Molecular Function being annotated is known to be intrinsic to the isolated gene product, annotate directly to the specific function in the Molecular Function ontology
  3. If the Molecular Function being annotated is known not to be, or is not known to be, intrinsic to the isolated gene product, annotate the component proteins with the Molecular Function of the entire complex and use the "contributes_to" modifier. This indicates that the gene product contributes to the function of the complex, but is not known to have the function by itself. To do this, click the "contributes to" checkbox as you create your Molecular Function annotation. The "contributes to" qualifier should only be used for Molecular Function annotations.

Exchange of DNA/RNA between species.

We will not curate GO from experiments where non-zebrafish DNA or RNA is introduced into mutant zebrafish in rescue-type experiments. This data belongs to the species who donated the introduced gene. However, we may curate GO in cases where zebrafish DNA/RNA is used in non-zebrafish cells or organisms. In this case the data belongs to the zebrafish gene. Curators should exercise caution however to avoid poor quality annotations. For example, if antisense was used to knock down expression of a gene in a xenopus egg, and then a zebrafish gene was injected to try and rescue the egg, I would not curate GO from this. The quality of the experiment is low from a curation point of view. How well was expression of the xenopus gene knocked down? Also note that conservation of Biological Process across species can sometimes be weak. Additionally, there are cases where the function of an ancestral gene has been split among the two copies of the gene when it is a duplicate in zebrafish. We should annotate function and process terms conservatively and cautiously when such a condition exists.

"Sensu" terms and term definitions.

The use of "sensu" does not limit usage of a term to specific species, but rather qualifies the term to mean "as in" a specific taxon. So for example "sensu Mammalia" means that the specific GO term is refering to something as it is known in the Mammals. If the Mammals and Zebrafish are the same with respect to the term in question, then usage of the term is fine. If Mammals and zebrafish differ with respect to the term, then a new term may be required. The ontologies have not been challenged with a non-mammalian vertebrate(ie. zebrafish). There may be areas in the graphs that we would like to annotate to, but which have been restricted by "sensu Mammalia" without an appropriate non-mammalian alternative. Bring these cases to Doug's attention as you encounter them and adjustments will be made to the ontologies to accommodate our needs if the biology in question is different between Mammals and Teleosts.

Creating new GO terms and modifying current ontology structures.

We request edits to the GO ontologies through the SourceForge GO Curator Requests Tracker, where they are evaluated and executed by people at the GO editorial office. Such requests can be submitted by any of us, however you should set up a SourceForge account before submitting requests. by Erik Segerdell, updated 2004/1.

Locus/allele registration

New mutant records are made in ZFIN in the following ways:

  • registration of a new locus and/or allele by a member of the research community
  • registration of a new locus and/or allele by a ZFIN curator, resulting from literature curation, personal communication, or data obtained through a mutant line submission to ZIRC
  • bulk upload from the research community

Submit a new mutant on the "Register Locus and/or Allele Designations" page. Before proceeding, be familiar with these guidelines:

  • ZFIN Policy for Registration of Locus and Allele Designations
  • Zebrafish Nomenclature Guidelines
  • Laboratory Designations for Mutant Alleles

Submissions are emailed automatically to the designated ZFIN "nomenclature guru" for approval and further processing.

Deficiency and Transgenic lines: We currently treat these as mutants in the data model. Fill the locus name field with the Df or Tg name according to the Nomenclature Guidelines, copy this name into the abbreviation field as well, enter a lab allele designation, and submit them as a new locus and allele.

"Unspecified" alleles: Sometimes it is necessary to create a placeholder allele for a particular locus because you need to link data (GO, FX, etc.) to that mutant, but an author has not provided an allele designation.

For this purpose, create a new allele called "un_[locus_abbreviation]", e.g. un_swr would be the "unspecified" allele of the swirl (swr) locus. Only one of these should be created for a given locus. In the Submitter's Comments field of the newly-created Fish page, paste the following:

This record has been created in support of data for which a publication has not specified an allele.

Un-named mutants: Hundreds of mutants recovered in the large-scale Tuebingen and MGH screens now exist in ZFIN with only a lab allele designation and no official locus name or abbreviation. Every now and then, another one will crop up somewhere—a pub, an email, a ZIRC submission—and will need to be added to ZFIN, following this example:

  • Locus: un-named m749
  • Locus abbreviation: unm m749
  • Allele: m749
  • Submitted as a new locus and new allele

Tuebingen allele nomenclature: Many Tuebingen allele names have an extra letter or two after the "t" and preceding the number, for example tq252 and tbx392. This is generally allowed. In other cases where an extra stray letter is found in an allele name, consult the nomenclature committee.

Consult a technologist if the registration interface does not accept any mutant that you believe is correct and appropriate for inclusion in ZFIN.

Curation of Locus and Fish records

A ZFIN Locus (ZDB-LOCUS) record provides an overview of the mutant, including the affected gene (if known), a summary of alleles, and a link to publications. The allele names are hot-linked to their corresponding Fish (ZDB-FISH) records. Information in the allele summary table is imported from the individual Fish records.

Linking Locus and Fish records

Occasionally a curator must change the locus with which an allele (Fish record) is associated, i.e. allele t999 used to be assigned to the tweedledee locus, but a paper now reports that it is allelic to the tweedledum locus. There is no curator interface for making this change. Ask a technologist to make the change, supplying the Locus and Fish ZDB-ID's of the fickle mutants.

Affected gene

If the mutated gene has been identified, go to the Locus page and update this field with a link to the corresponding gene record.

Attention! It is important to attribute this field accurately and properly. Identifying a mutant locus as a certain gene is taken seriously by the scientific community; not giving attribution where it should be or giving attribution where it shouldn't be will potentially create hard feelings towards ZFIN.

Attribution should be limited to the publication(s) that first reports the assignment of a gene to a locus. Do not include publications that report a new allele of an established locus. Multiple publications are possible under the following circumstances:

  • Multiple papers appearing back-to-back in a journal and both laying claim to linking the gene to the locus.
  • Multiple papers claiming to link a gene and a locus for the first time appear independently within a reasonable timespan—say, within 3 months of each other.
  • Multiple papers claiming to link a gene and a locus for the first time appear independently, and one paper states something to the effect of "While in press Soandso and Soandso reported that the bla locus corresponds to the suchandsuch gene".
  • At the discretion of the curators and the nomenclature committee in unusual cases where multiple mutant loci are reported, by different publications within a reasonable timespan, to be mutations in the same gene.

Previous names

"Previous names" of mutants can be of different forms and can appear in three different places in ZFIN. It can be a bit tricky trying to figure out exactly what should go where (and where they are going to magically show up later). General rules of thumb:

  • Locus: Previous mutant names are entered into the previous names field on the Locus page, i.e. dino was previously called chordino.

  • Fish: The previous names field near the top of the Fish page is used in cases where a mutant was once considered to belong to a different locus. For example, dinm52 was previously thought to be an allele of the captain hook locus and was later determined to be allelic to dino; in this field, you would enter captain hook (cptm52), using HTML superscript tags for the allele designation. Furthermore... these names will be imported automatically into the Locus page's previous names field; you should edit them only in the Fish record.

  • Allele: Previous laboratory allele designations are entered into the previous names field lower on the Fish page in the "ALLELE(S) CARRIED IN THIS MUTANT LINE" table, i.e. b104 was originally reported as b104rl.

Attribute previous names with the publication(s) in which the old names first appeared.

Mutant descriptions

If you are in edit mode of a Locus or Fish page, you may find descriptive paragraphs of the mutant and phenotype, along with some attributions. We currently hide these fields to the public and are not actively curating and updating them until we decide on a workable strategy for providing helpful, up-to-date, and complete text descriptions to the community.

The Locus page, however, still reveals to the public a short, one-line text description for each allele in the Brief Descriptions column of the allele table. Curators may continue to maintain this, making note of the phenotype and whether it is a strong, weak, or null allele, etc. See the ncad record for an example.

The alleles table is not directly editable on the Locus page. A little bit of magic is needed to modify the Brief Description lines. Go to the Fish record of the allele you want to edit. In the phenotype description field, type your brief description, as you want it to appear on the Locus page. Follow that with "Details:" enclosed by HTML bold tags:

Phenotype annotation

The keyword field on the Fish page displays a list of structures, developmental processes, and behaviors that are affected in the mutation. Keywords are based on a controlled vocabulary and are searched using the selector applet on the mutant search page.

For curation, use the keyword utility to generate a list of affected structures and defects in a mutant. Check the boxes next to each applicable term, copy the text string generated at the bottom of the page, and paste it into the ZFIN Fish record. Be sure to include the initial and final semicolons in the text string; the initial semicolon will be hidden when the keywords are displayed publicly.

Future developments: The current mutant keyword list is a limited set of terms, developed originally to annotate data for a large mutant screen nearly a decade ago. It will be replaced soon by PATO, the Phenotype and Trait Ontology, a cross-species controlled vocabulary used in conjunction with the anatomical ontology and gene ontology to annotate mutants. For now, curators should continue to use the old keywords, which will be translated to PATO terminology when its time comes.

If you do not find a term you need in the old vocabulary right now, you can copy and paste one or more items from the anatomical ontology into the keyword field, separating them with semicolons. They will become searchable terms once PATO is implemented. For example:

An existing Fish record has...

Keywords: nervous system;brain;small eyes;

A new publication describes specific defects in the retina and retinal pigmented epithelium of this mutant. The old keyword vocabulary does not contain terms for these; you can find approved terms in the anatomical ontology and append the keyword list:

Keywords: nervous system;brain;small eyes;retina;pigmented epithelium;

Morpholinos

This section was written (Mar. 2005, updated May 2005) by Doug Howe with nomenclature by Erik Segerdell

A CAUTIONARY STATEMENT

The quality of morpholino (MO) sequences reported in publications has been found to be highly variable. In addition, MO sequence is variably reported as either MO sequence or target sequence. We always curate the 5'-3' MO sequence into ZFIN. For this reason, we examine and evaluate morpholino sequences to ensure the highest quality and consistency of MO sequences possible prior to their addition into ZFIN. This portion of the DOC provides guidance on how to confirm an MO sequence and enter it into ZFIN.

Morpholino Nomenclature

Morpholino nomenclature follows the format:
MO#-xyz
... where "xyz" is the approved symbol of the targeted gene.

  • All names begin with "MO" (all caps)
  • "MO" is always followed by an ordinal to distinguish unique sequences targeting a particular gene. The first one added to ZFIN will be MO1 (not plain "MO"), then MO2, etc. This means that if an author refers to a morpholino as "pax8 MO1" and ZFIN already has five pax8 MO's that have different sequences, then the reported MO would be added to the database as "MO6-pax8", as all ZFIN records require a unique name. In this case "pax8 MO1" would be shown as a previous name of MO6-pax8 and would be attributed to the publication.
  • Additional information about the morpholino - i.e., how or where it targets the gene - is captured in its ZFIN record. We do not attempt to convey this info with additional symbols or notations in the name itself.
  • Curators and the Nomenclature Committee agreed that including lab designations in the name would significantly slow down and complicate curation, as in many cases we would have to arrange new designations for authors before an MO could be added to ZFIN. All MO records will list publicatons that show who has used it and how.
  • This naming format may be used for other RNA-blocking reagents - e.g., "PNA" in place of the "MO".
  • Use the ZFIN pseudopub ZDB-PUB-030508-1 for previous names not tied to a publication.

Confirming a morpholino sequence

Always confirm the MO sequence before adding it to ZFIN. You must confirm that the MO targets the correct gene, appears to work by the mechanism the authors claim is should, that it is not already in ZFIN, and that there is nothing suspicious about the sequence such as missing/extra bases etc. There are two ways to do this...

Confirmation by Local ZFIN BLAST

  1. GO to the Local ZFIN BLAST page.
  2. Paste the MO sequence into the sequence box and select the "Search for short, nearly exact matches" checkbox.
  3. Choose the desired databases (a good set includes ZFIN Morpholinos and GenBank Zebrafish or ZFIN cDNA)
  4. Set the Expect and word size values to 1000 and 7 respectively (on some browsers this will happen automatically)
Confirmation by NCBI BLAST
  1. Go to the page for NCBI BLAST for short exact matches
  2. Paste in the reported MO sequence.
  3. Select Danio rerio from the species filter pulldown. Word size and Expect values should by default be 7 and 1000 respectively.

Run the BLAST analysis and proceed as described below for translation or splice blocking morpholino sequence.

For translation blocking MOs

  1. Confirm that the MO targets the proper gene: Examine your BLAST results for a 100% full length blast hit on an mRNA sequence corresponding to the gene the authors claim the MO targets. If things go well, there should only be one gene with perfectly matching mRNA. The BLAST hit will commonly span the start codon of the mRNA. This means it will generally hit the 5' end of the mRNA. If you have the MO sequence, then the MO is likely to have "CAT" (the reverse complement of "ATG"), in the middle of it somewhere.
  2. Determine if you have MO or target sequence: If the Strand line of the BLAST results is "Strand = Plus / Minus" on an mRNA sequence match, then the sequence you BLASTed is complementary to the mRNA, so it is the actual MO sequence. If this line is "Strand = Plus / Plus" on an mRNA sequence match then the sequence you BLASTed is the same as the mRNA sequence, which means it is the target sequence rather than the MO sequence. This can be corrected when you enter the MO into ZFIN. If the sequence does not appear to be a translation blocking morpholino, contact the authors for confirmation

For splice blocking MOs

  1. Locate a genomic clone matching the published MO sequence: Look for a genomic sequence that has a 100% full length match with the sequence you BLASTed. Write down the accession number of the genomic segment and the bp coordinates that match the sequence you BLASTed.
  2. Confirm that the MO spans a splice junction: Get a full length mRNA for the targeted gene, and use this mRNA in a BLAST2 analysis with the genomic clone identified by your initial BLAST. You should find the targeted gene in the genomic clone. You should also find a segment of the BLAST2 hit that begins or ends within the same bp coordinates that were targeted by the MO.
  3. Determine if you have MO or target sequence: Analyze by BLAST2 the published "MO" sequence vs. the mRNA sequence. This will not be a full length match, but the portion of the MO sequence that corresponds to the exon will match. If the Strand line reads "Strand = Plus / Minus", then the published MO sequence is correct. If the Strand line reads "Strand = plus / plus" then the published MO sequence is actually the targeted mRNA sequence. This can be corrected when you enter the MO sequence into ZFIN. If the morpholino does not target the expected gene or does not appear to span an intron/exon junction, contact the authors for confirmation.

Adding a new MO to ZFIN

  1. GO to the ZFIN Add Marker Page.
  2. Select Morpholino from the pulldown menu and hit the "create it" button.
  3. Enter the MO name (see the MO nomenclature guidelines). (required)
  4. Copy and paste the sequence into the sequence box (limited to 100 bases). Manipulate the sequence if necessary using the adjacent buttons to get the MO sequence in 5'-3' orientation. Any non-ATGC bases will be dropped at this stage if you try to complement the sequence. A popup window will warn you of this.
  5. Enter the publication from which you are curating the MO if the pub is the first report of the MO sequence (optional)
  6. Enter any publicly viewable notes into the "Note" field below the name field. This could include a statement about the type of MO (splice or translation blocking) if you have confirmed this, as well as any information important for the public to view regarding the sequence. For example if you had to remove a base from the sequence of this MO as published in a specific paper, it would be reasonable to note that and acknowledge that the authors confirmed this edit.
  7. Enter any private curator notes into the "Curator Notes" field to the right of the sequence. This should include any additional information you may have uncovered while analyzing the MO sequence, communication with authors, etc.
  8. Submit the MO by hitting the "Submit new morpholino" button.
  9. Go to the new MO page and add any previous names (usually at least the one used in the pub you are curating) and attribute them to the proper publications.
  10. In the Marker Relationships section, add the "is morpholino of" relationship to the correct target gene, and add attribution for this as well.

Other morpholino considerations

  • You should be skeptical of claims in the literature that "our morpholino AGCTG....ATAAT is exactly the same as that used by" anyone else. It is not uncommon for the morpholinos to be shifted one way or the other by a few bases, but authors have been known to still consider these "exactly the same". For our purposes, a morpholino is defined by its sequence. Even a single base difference is a distinct MO in ZFIN. This should be checked out carefully, especially when it comes to back-curation of older pubs that are said to have used a specific MO.....check the sequence before adding the pub to a MO.
  • GeneTools, Inc. is currently the primary source for MOs. They have told us that they generally do not make MOs longer than 26 nucleotides because it messes up their production schedule. If a paper reports an MO longer than a 26mer, you should be skeptical.
  • If you have reservations or concerns about the MO sequence as it was reported in the paper, you can create the MO record without including the sequence. That way you can continue curation until the actual sequence has been clarified.
  • Sometimes there is no recourse but to contact the authors for confirmation.

Publications

This section was written by Sherry Giglia 7/2005

Pub Searches

At this time, PubMed is the primary source for locating recently published zebrafish papers. In the past, we checked other sources and subscribed to various pub alerts. We found that, for the most part, the information received was a duplication of what we had already received from PubMed and was not worth the time spent checking every paper against the database.

Exceptions:

  • PubMed does not currently carry papers published in the journal Zebrafish. However, this relatively new journal publishes papers by prominent zebrafish researchers, so it is important to capture this information. Because Zebrafish is a new journal (first published Feb., 2004), publication intervals have been sporadic. The AC will check the Zebrafish journal website a minimum of every 3 months to keep current with publications. This duty has been assigned to the student assistant in the past.

  • Papers published in fisheries journals and some toxicology journals do not always show up in PubMed. These papers come to our attention in various and sometimes unpredictable ways - usually at the request of authors. The rule of thumb is that a zebrafish paper will be entered into ZFIN, even if there is no "curatable" data in the article.

  • Books, PhD and Masters' theses about zebrafish. The AC or the student assistant should check periodically for these in World Cat (from the UO Library website).

To search for zebrafish papers in PubMed: PubMed must be accessed from the special UO url - not the public PubMed url.

http://www.ncbi.nlm.nih.gov/entrez/query.fcgi?otool=oruolib

This is important, because the UO url provides links to the UO library records of journal publications for each article, in addition to any journal webpage links provided by PubMed. This is very useful when entering pubs.

The search keyword is simply "zebrafish". This displays any paper with the word "zebrafish" in the title, abstract or keywords of the paper. Note that PubMed does not search the entire article. However, it is highly unlikely that a paper would be appropriate for entry into ZFIN if the word "zebrafish" did not exist in the title, abstract or keywords.

PubMed will list articles in publication date order. This is ideal for our purposes, because the most recent papers are listed first. The AC only needs to keep track of where she/he left off the last time pubs were entered to know where to begin the next time.

Entering New Publications

It works well to enter pubs 1-2 times per week. This has been the duty of the AC primarily in the past, but the student assistant is also trained to perform this task.

Steps for entering pubs:

  • Find the last pub entered, and start working with from the bottom up with new pubs (e.g. if pub #10 was the last one entered, begin your work with pub #9 and work up to pub #1). This way, if you are entering several articles from one journal issue, the papers with lower page numbers will be entered first, and therefore assigned a lower "ZDB-ID" number.

  • Click on the link for the first paper. If the word "zebrafish" does not occur in the title of the paper, do a "find" (control "f" in most browsers) to search for the word "zebrafish" in the abstract. You may be able to tell from the context of the usage of the word(s) whether or not this is, indeed, a zebrafish paper.

  • If you are not sure that this article is appropriate for ZFIN, start an e-mail to curators@zfin.org, and list the PubMed number of that paper. You can add to it as you go through the remaining pubs, if necessary. In this e-mail you will ask the curators to check the pub(s) and let you know if they want these pubs entered. If even one curator answers "yes", you need to enter the pub.

  • If you are sure that an article belongs in ZFIN, open the ZFIN website in a new screen (this is where the two monitors come in very handy!) From the home page, click on "Add new PUB" under "Publications" to access the entry form.

  • Copy and paste all appropriate information from PubMed to ZFIN:
    • TITLE: Do not copy the period at the end of the title - ZFIN will add one when you submit the new pub record

    • STATUS: When they first appear in PubMed, most publications are still "Epub ahead of print," which is the default value. You will see these words in the citation of the paper in ZFIN. Other status options:
      • Active: If the publication date is current (not more than a week ahead of the current date), and volume, issue and page numbers are listed in PubMed, this is an "active" pub.
      • Inactive: If volume, issue and page numbers exist, but the publication date is in the future, this is an "inactive" pub - it has probably not been published yet, but is not considered an Epub.
      • In press: Rarely used - usually refers to a book

    • PubMed ID: Copy and paste from the PubMed record (at the bottom of the abstract).

    • AUTHORS: Copy and paste from the PubMed record. Some additional editing is required. PubMed does not place commas or periods in author listings. You must manually place a comma after each last name, and a period after each initial. Additionally, you must add the word "and" before the last author's name (look at other pub records for examples).

    • NUMBER of authors listed: Add 'em up!

    • DATE: Date as shown in the PubMed record. Note: If only the month is listed, use the 1st as the date (e.g. 7/1/05 for a July issue). If only the year is listed, investigate a little further. If you can find the article on the journal website, you may be able to determine the month of publication. If not, use January 1st of that year.

    • Journal Abbrev: This abbreviation must match the journal record that exists in ZFIN. Abbreviations are almost always the same as what is listed in PubMed, except for the fact that PubMed does not use periods for truncated words in the title. You must add these.
      Example:
      PubMed shows:Dev Biol
      You Use:Dev. Biol.

      To be sure, you can click on the "find journal abbrev." button and find the right abbreviation. If you don't look it up and the abbreviation you use is wrong, you will get a cryptic error message when you submit the pub record. Not to worry - you can simply hit the back button and try again without losing all your data.

      If ZFIN does not have a record for the journal, you must create one. It works best to open another window (or tab) and go to the ZFIN home page to access the "Add new JOURNAL" form, under "Publications". To create this record, you must determine the full name of the journal. The best way to do this is to click on the "FindText" link on the PubMed article record. This takes you to the UO Library record, where the full name of the journal is listed. To obtain the correct journal abbreviation, you are usually safe to use PubMed's abbreviation and add the periods. Be careful about short words that may or may not be abbreviated (e.g. "cell" which could actually be the word "cell" and would not require a period, or could be short for "cellular".)

    • Journal Volume: Volume number and issue number in this format:
           426(6)
      Do not add the colon! ZFIN will add it when you submit.

      NOTE: If you are working with an "Epub", you will not have the volume or page numbers - leave them blank for now.

    • Pages: Page numbers, written out. Note that PubMed abbreviates this, but ZFIN does not.
      Example:
      PubMed shows:145-7
      You Use:145-147

    • Type of publication: Usually "journal" which means "journal article". The other most common pub type is "review". A good clue that an article is a review is if there is only one author. Also, the title often gives it away. Check the journal website to try to determine if an article is a review if you are not sure. Often, you can find that designation in the table of contents (not in the article page itself).

    • KEYWORDS: PubMed does not list keywords. You will only find these if you can find the article or the abstract online. There is usually a link to the online version of the article or abstract in the PubMed record, or sometimes you can find it through the "FindText" UO link. However, even if you do find the article online, it will not always have keywords. It is worthwhile to try to find them, however, and if you do, copy and paste. NOTE: if keywords are separated by anything other than commas or semicolons, you must change the separator. Other punctuation may cause an error response when you submit.

    • ABSTRACT: Copy and paste this from the PubMed record - not from the journal website record. The reason for this is that PubMed will never use scientific symbols in their abstracts, and the journal websites often do.

    • ERRATA and NOTES: This field is used when an erratum is published to correct or change information in a previously published paper. PubMed will display a new record with an erratum that refers to the original paper. ZFIN does not create a separate record for the erratum; the information is copied and pasted into this field in the original pub record. Be sure to include the new PubMed number for the erratum for reference.

      This field is also occasionally used for notes about the record, e.g. "article published in Chinese" or any other useful information. Keep in mind that this field will show up in the pub record when it is queried, so it should not be used for "internal" notes.


  • Click the "submit" button to create the new pub record.

Linking to Authors

After a pub record has been created, the paper must be linked to author(s) that have personal records in ZFIN. Once linked, the paper becomes part of each author's record and the lab record with which they are affiliated.

To link papers to authors:

  1. Click on the "link" button to the left of the reference for the new record. A javascript form will be displayed, showing all names that are possible matches to the author names listed in the pub record.

  2. Once you have identified a correct author name in the "potential authors" list, click on it and then click on the ">>>" arrow button to move this name to the "linked author list". Repeat this process for each author found in the "potential" list.

  3. Click "submit" to complete the link function.
Note: Sometimes the "potential authors" list will be blank. This just means that none of the authors names are at all similar to those who have records in ZFIN.

Tips for linking authors:

  • Always check author names in the online journal article or abstract, where the full first name is usually displayed (PubMed uses only initials).

  • If you are still unsure whether or not you have the right person, look at the institution that the author is affiliated with (either in PubMed or the online journal record) Then check this person's ZFIN personal record to see if it matches up.

  • Another clue is to look at who a particular researcher has published with in the past. Authors tend to publish with the same people multiple times.

  • If you cannot be sure, it is best to NOT link the author and contact the person you suspect is the author and check with them. Researchers don't like being associated with papers they did not publish, nor do they like having another person's name on their paper!

  • A word about special characters: Because we deal with a global network of researchers, you will frequently run across ethnic names with different accent marks, etc. ZFIN's policy is to include these special characters in a users personal record. PubMed, however, does not include these characters in their author names. If you have created a record that does not include a special character in an author name, and his/her ZFIN record includes the character, the name will not appear in the "potential authors" list. There are two ways to deal with this:

    • Watch carefully for special characters in author names in the online journal article or abstract. If the paper was published with the special character in the author's name, then you should copy and paste the last name into the ZFIN pub record. Then, it will show up in the "potential author" list and you can link it.

    • If the paper was published without the special character in the author name, you should not include it in the pub record. But if you know that the author's ZFIN record has a special character, you can link the pub from the author's personal record. To do this, pull up the personal record, go into update mode and click on "link new pub". Search for the pub in question and "select" it. It will now be linked to the author and lab record. Note that you are not likely to catch this until you have worked with these records for awhile and are familiar with author names. More commonly, the author will e-mail you and ask you to add a recent publication to his/her record. Upon investigation, you will realize that the reason the link was not made was because of the special character discrepancy.

Linking Pubs to New Personal Records

When a new personal and/or lab record is created for a PI or published zebrafish researcher, it is likely that articles authored by them already exist in ZFIN. The AC should check for pub records that should be linked to this new personal record from the personal record update screen, as described above.

Working with "Epub" and "Inactive" Records

At least once every week or so, the AC works the "Epub ahead of print" and "Inactive" lists. This should be done BEFORE adding new pubs; otherwise, you will have a longer list to work through because your list will contain the records you just entered.

To work the "Epub ahead of print" list:

  • Search for all Epubs by choosing "Epub ahead of print" under "Publication Status" on the publication search page. This will generate a listing of all Epubs in alphabetical order by first author name.

  • Open PubMed in a separate screen.

  • Click on the first paper title to bring up the pub record.

  • Copy and paste the PubMed number into the PubMed search bar to bring up the PubMed record. (Note - you could just click on the PubMed number in the ZFIN Pub record, but it makes navigation back to your working list more difficult Besides, it's easier to work on two monitors for this task).

  • Check the current status of the paper. If it still says "Epub..." in PubMed, go back to your ZFIN screen, hit the back button and go on to the next pub on the list. If the "Epub..." notation has been replaced with a new date, volume/issue and page numbers, go back to your ZFIN list, click on the "edit" link and change/add the following:
    • Status - change to "active" (or "inactive" if the new date is in the future)
    • Date - insert new date as displayed in PubMed
    • Journal Volume(issue)
    • Pages

  • Also check for keywords. If they are missing, it is worth your time to try to locate them again. Epubs are not always available online, and therefore you may have been unable to locate the keywords at the time the pub record was created.

  • Submit the record, which takes you back to your working list (notice that the pub you just submitted is gone - it is no longer an Epub!)

  • Repeat this process with each record on the list

To work the "Inactive" list:

  • Search for inactive records in the same way you searched for Epub records.

  • For this list, it is not usually necessary to refer back to PubMed. Simply click on the "edit" button for each pub record and look at the date.

  • If the date is current, change the status to "active" and submit the record, which takes you back to your working list.

  • If the date is still too far in the future, click "cancel," which takes you back to the working list, and go on to the next pub record.

NOTE: An e-mail is generated listing all the "active" pubs entered in a day and sent to curators and to the student assistant. This list includes any new active pubs entered, as well as any older pub records whose status was changed to "active" that day. The student assistant then locates a pdf version of each pub if possible, downloads it to the pub record for curation purposes and prints a hard copy, or tracks down a hard copy at the science library if a pdf is not available. She then makes the hard copies available for curators.

When the Web Pages Aren't Enough

The ZFIN web pages support the entry of most new data. However, they are not so good at supporting the modification of data that is already in the database. Frequently, a curator will change the name of something through the web interface and then find that the name has changed in some, but not all places in the web site. We plan to eventually fix these problems, but in the meantime many such updates require additional work by a technologist.

Whenever a biologist does any of the following types of updates they should notify a technologist. You should provide the ZDB IDs of the records involved as well as a description of what was done:

  • Deleting almost anything.
  • Renaming almost anything.
  • Merging almost anything.
  • Splitting almost anything.

As a technologist, what are you supposed to do with this information? Basically, you need to clean up the database, by deleting, renaming, splitting or merging any data that the web interface did not take care of. You will probably want to use the scancharcols.pl script to support this.

This document is one of several that describe the Zebrafish Information Network, or ZFIN. This document introduces ZFIN and the rest of the documentation.

ZFIN

ZFIN is an acronym for the Zebrafish Information Network, a web site of information about zebrafish for biologists around the world. The site contains all manner of information about zebrafish, from care and feeding, to where a particular gene has been mapped on which chromosome using which mapping technique.

Some of the information, such as care and feeding, is stored in static pages. Other information, such as genetic mapping results, are stored in a database and pages describing that information are generated dynamically in response to user queries.

ZFIN Documentation

You are reading the "ZFIN Documentation", such as it is. It consists of a hierarchy of HTML files. Our goal is to eventually include all documents about ZFIN in this HTML hierarchy. However, not all documentation is (or should be) in this tree. Much of the documentation about the implementation is stored in the web site files themselves. There is a lot of documentation in the makefiles that are used to produce the web site. This is in keeping with the principle that it is best to store documentation about code as close to the code itself as possible. This increases the likelihood that the documentation will stay up to date. Another excellent source of documentation for ZFIN is FogBUGZ, a project management / defect tracking tool used at ZFIN. See the FogBUGZ section for more details.

The documentation is split into a few large files, rather than one monolithic file, or hundreds of small files. This middle approach is a tradeoff between the advantages and disadvantages of either extreme. Having a few large files instead of hundreds of smaller ones makes it easier to search and to print the documentation. Having a few large files broken down along functional lines, rather than one large file will hopefully make the documentation easier to read.

How To Update This Documentation

There are 2 main ways to update this documentation. Which one you use depends on how comfortable you are with Unix, and how much you know about CVS. This discussion assumes that biologists are not comfortable with Unix and CVS and that technologists are.

Biologists

How the update is done depends on which section you are updating.
Curation To update the curation document, pull the document into whatever web page authoring tool you are used to (e.g. FrontPage, DreamWeaver) and edit it. When you are done with your updates, e-mail the updated page to a technologist (probably Dave C.), and they will put it back in to the documentation.
All other sections Same as for curation except for the last step. When you e-mail the updated file to a technologist you should also include a list of what sections were changed. Unlike for the curation document, the technologist will not do a wholesale replacement of the file with the updated file. Web page authoring tools are notorious for converting readable simple HTML files into obscure, over-tagged, and difficult to decipher HTML files. The curation document will be updated mainly be people using such tools. However, all the other documents will be mostly updated by technologists working directly in HTML, and we want to keep those files as clean as possible.

Technologists

Technologists use the same process to update all the HTML files. The documentation is part of the Commons CVS project. To edit documentation files, first checkout the Commons project, edit the files, and then commit them. Finally, talk to Dave C. to get the updates posted back to the ZFIN Commons directory.

Figures

The figures in this documentation were created with the Unix xfig utility. xfig is a drawing program capable of exporting files into a number of formats, including GIF, which is what the figures you see in these documents are. Since the figures were created with a Unix utility, they can only be updated by people who are comfortable with Unix, which means technologists. The xfig man page and the Using XFig manual are good sources of information on how to use xfig.

The figures are stored in subdirectories hung off the main directory. Each HTML document has its own subdirectory for figures. The versions of the files that are displayed in the doc end in .gif. The versions of the files that can be edited with xfig end in .fig.

The .fig and .gif files are controlled by CVS, just like the HTML files. The process for updating them is the same as that for HTML files, with a little extra work:

  1. Using CVS, checkout the Commons project (if you haven't already done so).
  2. CVS edit both the .fig and .gif versions of the figure file you want to update.
  3. Update the .fig file using xfig.
  4. CVS commit the .fig file.
  5. Run xfig on the committed .fig file, and Export it as a GIF, with Smooth turned on, and magnification set to at least 125%.
  6. CVS commit the .gif file.
  7. Talk to Dave C. to get the committed files posted to the ZFIN Commons directories.

The .fig and .gif files need to be committed in two separate steps because most of the .fig files contain RCS flags that show the revision number of the figure. Committing a file changes its revision number. Committing them separately means the revision number shown in the .gif file will always agree with that in the .fig file.

The figures were created with these settings:

  • Grid Mode set to the finest level.
  • Text Size set to 8
  • Test Step set to 1.0
  • Text Font set to Helvetica

A Zoom of 1.5 works well when editing the figures.

The GIFs were created from the xfig drawings by exporting them:

  • Select Export from the File menu.
  • Set Language to GIF.
  • Select Smooth.
  • Set Magnif to at least 125%. Bigger is better. Anything smaller will not be readable as a GIF. The diagrams you see in these documents were created with magnifications of 125% or 140%.
  • Hit the Export button at the bottom.

Database Documentation

The database documentation is divided into 2 different parts:

  • db.html: This contains database wide documentation, such as descriptions of ZDB ID formats, source versus data issues, and database functions. It also contains a pointer to the ER/Studio database documentation.
  • ER/Studio database documentation: This contains ER diagrams of the entire database and subareas of the database, and detailed descriptions of tables and columns.

The db.html file is updated in the same way as other ZFIN documentation. The ER/Studio doc is updated in an entirely different fashion, and the rest of this section deals with how to do that.

Note: This section does not talk about how to update documentation within the ER/Studio product itself. For details on how to do that, see FogBUGZ case 46. This section describes how to generate HTML reports from ER/Studio and then get them placed so they can be read from the ZFIN web site.

  1. First, in order to even do this, you must have a PC that has ER/Studio on it. As of 2004/01, Dave and Sierra's PCs have ER/Studio, and the FogBUGZ server in room 333 also has ER/Studio.

  2. In order to generate the Reports in the right place you must first set the Reports directory. In ER/Studio, get to the Options page:

      Tools → Options → Directories

    Set the "Intranet Site Root" to an Acme Network Place that contains a checked out copy of the Commons CVS project. It should point at the "doc/Database/DataModel" subdirectory. For example, on Dave's machine the "Intranet Site Root" is set to

    \\acme.cs.uoregon.edu\research\zusers\clements\Zfin\Commons\doc\Database\DataModel

  3. To generate a report of the production model, first open the production model in ER/Studio. The production model is at:

    Commons/doc/Database/ERStudio/Reference/Production.DM1

  4. Then generate a report:

      Tools → Generate Reports ...

    Under the "Wizard Quick Launch" section hit the "Load File..." button and set it to

    Commons/doc/Database/ERStudio/Reports/FullHTML.rpo

    Then hit the "Next" button. This may come up and warn you that the directory already exists. This will be the case if you have generated a report before. Press "Yes".

  5. The next page in the ER/Studio Report Wizard (page 2 of 4) lists what objects from the database you want to include in the report, and what you want to know about them. The selections you see here came from the "Wizard Quick Launch" file you selected on the previous page. Accept them as is and press "Next".

  6. The next page in the ER/Studio Report Wizard (page 3 of 4) lists what quality and zoom you want for the images, and what the logo, report title and description for the report should be. I spent a fair amount of time trying to figure out what minimum zoom (80%) and quality (100%) settings could be used and still have the diagrams be readable. I have also tried to add a Description here, but if I then save that back to the Quick Launch File, it corrupts the file. So you can add a description here, but think twice (or thrice) before saving it. Accept the settings on this page and hit "Next".

  7. The fourth and final page basically lists what will show up on the title page of the report. I would really like to be able to set the Modified date here, but I can't figure out how. I am not sure if the version is useful to us. Perhaps we should set it to the date? Anyway, accept the settings on this page, and hit "Finish".

  8. This then displays a window showing the progress of the report generation. When the generating is complete the "Cancel" button changes to "Close" and ER/Studio opens a browser window with the report in it. Close the window and then review the report in the browser to see if it is what you want. If it is then proceed to the next step. If it isn't then go back into ER/Studio and either update the data model (or the report options) and then regenerate the report.

  9. You now have an HTML report that exists on your Unix filesystem in your checked out copy of the Commons directory. You now need to copy that report to the "production" and "development" copies of the Commons directory so the report is visible on the web. The previous instructions were all run on your PC. The remaining instructions are all run on Unix.

  10. Get to the DataModel directory of your copy of the CVS Commons project. For me this is

    % cd Commons/doc/Database/DataModel

  11. The DataModel directory is a mixture of CVS controlled files (README) and files that are generated by ER/Studio. The files that are generated by ER/Studio are not placed under CVS. When ER/Studio generates the files and directories under the DataModel directory, it does not always make them world readable. We want everything to be world readable:
    % find . -exec chmod g+r,o+r {} \; % find . -type d -exec chmod o+x {} \;
  12. Package the directory into a tar file.
    % cd .. % tar cf DataModel.tar DataModel
  13. Copy the new report into the "development" ZFIN commons directory so it can be seen from the ZFIN documentation. As yourself on the development server:
    % sudo cp DataModel.tar /private/ZfinLinks/Commons/doc/Database % cd /private/ZfinLinks/Commons/doc/Database % sudo rm -rf DataModel % sudo tar xf DataModel.tar % sudo rm DataModel.tar
  14. Copy the new report into the "production" ZFIN commons directory so it can be seen from the zfin.org web site. As yourself on the production server:
    % sudo cp DataModel.tar /private/ZfinLinks/Commons/doc/Database % cd /private/ZfinLinks/Commons/doc/Database % sudo rm -rf DataModel % sudo tar xf DataModel.tar % sudo rm DataModel.tar

And you are done.

ZFIN Web Sites

The end product of ZFIN is a web site. In most contexts that means the production site, http://zfin.org. However, ZFIN also has several other types of sites for things like mirror sites, beta testing, and development. The remainder of this section briefly introduces each type of ZFIN web site. For information on what web sites are implemented on which machines using which databases see the Web Site / Database / Machine Matrix.

Production Site

When people talk about the ZFIN site, they are referring to the production ZFIN site at http://zfin.org. This is ZFIN's presentation to the world. It has the latest and greatest version of the web site. The biologists do their curation on this web site. No matter which physical server the production web site is running on, it is always accessed as http://zfin.org.

The production web site normally resides on the production server. However, when the production server is down for upgrades or maintenance, the production web site resides on the development server. The process for doing this swap is described in the Moving zfin.org to a Different Machine section.

Pre-Production and Sandbox Site

ZFIN uses the http://almost.zfin.org site as its final staging/testing site for updates before they are placed into production. ZFIN team members will often be asked to test something on this site as a final sanity check. It also serves as a "sandbox" site where curators can try things before trying them on http://zfin.org to see what happens.

Beta Testing Site

ZFIN sometimes makes new features available to the outside world for testing and evaluation before we put them into production. We use the http://test.zfin.org site for this purpose. Access to the testing site is generally disallowed, but is opened up when features are being tested.

In addition, we also sometimes open up access to particular development web sites that have features we want people outside of ZFIN to evaluate.

Development Sites

ZFIN has a score of development web sites that are used to do ZFIN maintenance and development. They are named http://mutant_name.zfin.org. Most development sites are backed by their own database. See the Web Site / Database / Machine Matrix for a list of development sites.
